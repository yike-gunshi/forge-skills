# v2026.04.25 — Bug 修复验收报告与 QA 自动闭环

本次发布把 `forge-qa` 与 `forge-bugfix` 从“测试后再修 bug”的松散协作，升级为一套可跨会话、可批量、可人工最终验收的闭环体系。

## 核心理念

- **批次 = 管理单元**：聚合多个 bug 的状态、进度、最终验收入口。
- **Bug = 工程单元**：每个 bug 都有独立编号、独立报告、独立状态。
- **Worktree = 隔离单元**：每个 bug 独立 worktree，失败可丢弃，合并可预演。
- **TDD = 质量单元**：每个 bug 都要有 RED 失败证明和 GREEN 通过证明。
- **QA = 验收单元**：每个 bug 都由 forge-qa 回归，最终由用户或批次验收收口。

批量修复只改变管理方式，不改变单 bug 的工程边界。

## 新能力

### Bug 修复验收报告

单 bug 文档从“修完后生成的验收清单”升级为从发现到关闭的案卷：

```text
docs/bugfix/reviews/BF-XXXX.md
```

报告记录：

- 问题发现记录、初始截图和关联 Feature Spec
- 复现步骤和根因分析
- worktree、branch、commit、涉及文件
- TDD / 回归用例的红绿证据
- 前后端验收地址和环境身份校验
- 人工验收指南
- QA 测试过程、逐步截图和深度断言
- 最终结论和新发现分流

### QA 自动闭环

功能开发完成后，`forge-qa` 可以发现结构化 bug 候选，交给 `forge-bugfix` 自动修复，再由 `forge-qa` 回归：

```text
forge-qa
→ BF candidate
→ forge-bugfix 独立 worktree/TDD 修复
→ forge-qa Mode B 回归
→ qa-pass-pending-final-review
→ 最终人工验收
```

自动闭环不是无限循环。同一 bug 连续回修失败 3 次，或遇到需求、设计、环境身份疑问时，必须进入 `blocked-human`，由用户判断。

### Codex Browser Use 接入

在 Codex 环境中，前端页面、交互、视觉、控制台相关 QA 优先使用 `browser-use:browser`：

- 用 Codex in-app browser 打开本地 `APP_URL`
- 采集 DOM snapshot 和用户视角截图
- 将关键状态节点截图保存到报告并 Markdown 内嵌
- 读取 console logs，`console.error` 进入 FAIL

Playwright 继续负责可重复脚本断言；gstack/browse 可作为已有项目补充；Computer Use 只在 Browser Use 不可用或目标不是浏览器页面时兜底。

### 前后端环境身份强校验

QA 和人工验收必须使用同一组地址。报告会展示：

- Frontend URL
- Backend/API URL
- URL 来源
- PID
- cwd
- branch / commit
- health check 或关键接口探活结果

如果 PID/cwd 不属于当前 worktree，或 QA 实际访问地址与人工验收入口不一致，直接 QA_FAIL。

### v7.1 证据完整性门禁补充

本 release 追加 `forge-bugfix` v7.1，进一步把验收质量从“AI 描述已修复”收紧到“用户问题闭环已被证据证明”：

- 新增 `validate-bugfix-report.py`，在进入人工验收前机器校验报告。
- 前端/交互 bug 必须有 Markdown 内嵌 QA 截图，且截图文件真实存在。
- 报告必须包含“用户问题闭环断言”：用户原话、最终用户可见结果、证明方式、PASS/FAIL 结论。
- 涉及前端展示、权限边界、数据可见性、灰度/冷启动的 bug，必须记录配置 / 开关 / 数据就绪检查。
- 每个 bug 原则上必须提供独立 worktree 证据；无独立 worktree 不进入 P5。
- 状态枚举收紧为固定集合，`qa-incomplete` 不能被描述成 pending user verification。

### v7.2 Pass 边界与接力 prompt

本 release 继续追加 `forge-bugfix` v7.2，把“原 bug 已经修完”和“发现了下一个问题”明确切开：

- 原 bug 一旦 Pass，AI 必须先告知用户本次 bug 修复已完成。
- 不论新问题来自用户反馈、forge-qa、批次验收，还是 AI 发现的衍生问题，都不得直接继续修。
- 新问题写入 `docs/bugfix/backlog.md` 或报告索引前，必须先询问用户是否确认入档。
- 用户确认入档后，AI 给一段简短接力 prompt，建议用户开另一个会话继续，并让新会话重新判断范围和根因。

## 变更内容

- `forge-bugfix` 新增 P2.5：bug 登记/领取时创建或更新 Bug 修复验收报告。
- `forge-bugfix` P5 改为独立 TDD 驱动修复，P5.3 写入红绿证据。
- `forge-bugfix` P6 支持单 bug 验收和批次待最终验收两种节奏。
- `forge-bugfix` v7.1 新增用户问题闭环、配置/开关/数据就绪和报告完整性校验器门禁。
- `forge-bugfix` v7.2 新增 P7.4 Pass 边界确认和简短接力 prompt。
- `forge-qa` Mode A 发现问题时产出结构化 BF candidate。
- `forge-qa` Mode B 回填 Bug 修复验收报告中的 QA 证据区、环境身份区和逐步截图。
- `backlog.md` 增加 `fixed-awaiting-qa`、`qa-failed`、`qa-pass-pending-final-review`、`blocked-human`、`deferred` 等状态。
- `review-checklist.md` 模板重写为 Bug 修复验收报告模板。

## 适用场景

### 小批量修复

一个会话内完成：

```text
QA → bugfix → QA 回归 → 用户验收
```

### 大系统多天修复

多个会话并行处理：

```text
backlog / QA 报告
→ 多个 BF 分别领取
→ 每个 BF 独立 worktree/TDD/QA
→ 批次汇总
→ 用户最终统一验收
```
