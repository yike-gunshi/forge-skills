# 技术调研报告 — v2026.04.24-fupan-workbench

## 调研时间

2026-04-24

## 调研范围

为 `forge-fupan` 增加本地 Fupan Workbench：FastAPI 本地服务、React/Vite 静态前端、task JSON 状态流、历史复盘 Markdown 浏览、`forge-fupan` 启动/轮询/降级流程。

## 一、技术方案推荐

### 1. 推荐总体方案

采用“skill 内嵌轻量 fullstack”：

```text
skills/forge-fupan/
├── SKILL.md
├── parse_tokens.py
└── workbench/
    ├── README.md
    ├── requirements.txt
    ├── launcher.py
    ├── server.py
    ├── task_store.py
    ├── review_index.py
    ├── markdown_reader.py
    ├── static/              # Vite build 产物，运行态优先托管这里
    ├── frontend/            # React + Vite 源码
    └── tests/
```

这样不把 `package.json`、`requirements.txt` 放到 forge-cookbook 根目录，避免把 skill 仓库误识别成普通前后端项目。

### 2. 后端

- 使用 FastAPI 提供 API 和静态文件托管。
- 使用 Uvicorn 启动本地 ASGI 服务。
- 只监听 `127.0.0.1`，不得监听 `0.0.0.0`。
- API 前缀使用 `/api/*`，前端路由使用 `/` 和 `/review/{review_id}`。
- 静态前端使用 FastAPI `StaticFiles` 托管。FastAPI 官方文档说明 `StaticFiles` 可从目录自动服务 JS/CSS/图片等静态文件，并通过 `app.mount()` 挂载到指定路径。

参考：
- FastAPI Static Files: https://fastapi.tiangolo.com/tutorial/static-files/
- Uvicorn Settings: https://www.uvicorn.org/settings/

### 3. 前端

- 使用 React + Vite。
- 运行时不启动 Vite dev server，而是托管已构建的 `static/`。
- Vite 官方静态部署文档说明默认 build 输出目录是 `dist`，可通过 `npm run build` 生成后部署；`vite preview` 只用于本地预览，不应作为生产服务。
- 本项目应把 `frontend/dist` 同步到 `workbench/static`，由 FastAPI 服务。

参考：
- Vite Getting Started: https://vite.dev/guide/
- Vite Static Deploy: https://vite.dev/guide/static-deploy.html
- Vite Backend Integration: https://vite.dev/guide/backend-integration.html

### 4. React 状态

- 当前展开 task、任务队列、topic selection、提交状态应由顶层组件集中管理。
- React 官方文档建议：需要协调多个组件时，把状态提升到共同父组件，形成单一事实源。
- 表单提交使用标准 `<form>` 语义，并提供 submitting / submitted / failed 状态。

参考：
- React Sharing State: https://react.dev/learn/sharing-state-between-components
- React form: https://react.dev/reference/react-dom/components/form

### 5. 依赖策略

第一版依赖应克制：

后端 `requirements.txt`：

```text
fastapi
uvicorn
httpx
pytest
```

说明：

- `httpx` 是 FastAPI TestClient 的前置依赖之一，FastAPI 官方测试文档也提示使用 TestClient 前需安装 `httpx`。
- `pytest` 只用于本地测试，不影响运行态。
- Markdown/frontmatter 优先用标准库实现，避免 PyYAML 等额外依赖。

前端 `package.json`：

```text
react
react-dom
vite
@vitejs/plugin-react
typescript 或纯 JS
marked 或 react-markdown
```

建议第一版使用纯 JS + `marked`，减少 TypeScript 配置成本；如果后续需要更严格维护，再升级 TS。

## 二、架构模式推荐

### 1. 服务启动与复用

推荐 `launcher.py` 提供 CLI：

```bash
python3 skills/forge-fupan/workbench/launcher.py start --task-id <task_id> --open
python3 skills/forge-fupan/workbench/launcher.py create-task --input <learning_map.json>
python3 skills/forge-fupan/workbench/launcher.py wait --task-id <task_id> --timeout 1800
```

状态文件：

```text
~/.forge/fupan-workbench/server.json
```

字段：

- `host`
- `port`
- `pid`
- `url`
- `started_at`
- `repo_path`
- `skill_path`

复用逻辑：

1. 读取 `server.json`。
2. 请求 `GET /api/health`。
3. 健康且版本兼容则复用。
4. 不健康则选择新端口启动。
5. 启动后写回状态文件。

### 2. Task 存储

推荐 task 文件放在：

```text
~/.forge/fupan-workbench/tasks/{task_id}.json
```

原因：

- 与复盘 Markdown 归档目录解耦，避免临时 task 污染长期知识库。
- 所有项目共用一个 workbench 任务队列，便于多项目首页展示。
- `forge-fupan` 只需记住自己的 `task_id`。

写入策略：

- 每个 task 独立 JSON 文件。
- 写入时先写 `{task_id}.json.tmp`，再 `os.replace()` 原子替换。
- `submitted` 后保留原 selection，不允许重复覆盖；如果用户需要改，下一版再做“重新打开选择”。

### 3. Review 历史索引

历史复盘仍以 Markdown 为真相源：

```text
~/claudecode_workspace/记录/复盘/{项目名}/
```

`review_index.py` 负责扫描：

1. 全局 `INDEX.md`，优先解析标题、项目、标签、摘要。
2. 单项目 `INDEX.md`，作为补充。
3. 无索引时扫描 Markdown 文件名、mtime 和 frontmatter。

前端历史列表字段只使用：

- 日期
- 标题
- 项目
- 学到的知识 `learned_topics`

不得展示 selection、depth、内部 JSON。

### 4. API 设计

建议第一版 API：

| Method | Path | 用途 |
|---|---|---|
| GET | `/api/health` | 服务健康检查 |
| GET | `/api/tasks` | 任务队列 |
| GET | `/api/tasks/{task_id}` | 单个 task |
| POST | `/api/tasks/{task_id}/selection` | 提交 selection |
| GET | `/api/reviews` | 历史复盘索引 |
| GET | `/api/reviews/{review_id}` | Markdown 详情 |

关键约束：

- review 详情只接受 `review_id`，不得接受任意文件路径。
- 后端用 index 映射 id 到真实 path，并检查 path 位于复盘根目录下。
- task 只允许状态机合法流转：`pending_selection -> submitted -> consumed/failed`。

## 三、风险与坑点

### 1. 依赖缺失

风险：用户没有安装 FastAPI/Uvicorn，触发复盘时服务启动失败。

处理：

- `launcher.py` 捕获 ImportError。
- 输出可读错误和安装命令。
- `forge-fupan` 降级到对话内确认。

### 2. 端口与进程混淆

风险：旧服务占端口，用户打开的是旧版本页面。

处理：

- 不猜常见端口。
- `server.json` + `/api/health` 必须返回 `skill_path`、`repo_path`、版本号。
- 若健康检查路径不匹配，启动新服务。
- 启动 URL 必须来自 launcher 输出，而不是硬编码。

### 3. 多任务串读

风险：A 会话创建 task A，B 会话提交 task B，A 误消费 B。

处理：

- `forge-fupan` 轮询时必须传 `task_id`。
- `wait` 命令只读该 task 文件。
- `submitted` payload 中保留 `session` 和 `source_thread`，用于额外防错。

### 4. SPA 路由与静态托管

风险：访问 `/review/{review_id}` 时 FastAPI 静态文件 404。

处理：

- 后端为非 `/api` 路径回退返回 `index.html`。
- 静态资源仍由 `/assets/*` 或 build 输出路径直接服务。

### 5. Markdown 安全与可读性

风险：历史复盘 Markdown 中可能有 HTML、路径、长表格。

处理：

- 第一版前端渲染 Markdown 时默认不执行脚本。
- 源文件路径只作为文本或安全链接展示。
- 表格容器横向滚动。
- 长代码块保留滚动。

### 6. 构建产物漂移

风险：修改 React 源码后忘记重新 build，运行态仍是旧页面。

处理：

- `npm run build` 后同步 `frontend/dist` 到 `workbench/static`。
- QA 必须用 FastAPI 实际托管 URL 截图验证。
- 文档中记录“源码变更必须带构建产物变更”。

### 7. 当前仓库脏工作区

当前 `main` 上已有多处未提交 skill 文件改动，加上本次新增 PRD/DESIGN/CONTEXT/RESEARCH。

处理建议：

- 开始工程实现前，先明确是否在当前 `main` 直接继续，还是创建隔离 worktree。
- 由于 PRD/DESIGN/CONTEXT/RESEARCH 目前是未提交文件，若创建 worktree，需要先把这些文档以 patch 或拷贝方式带入 worktree，不能用 destructive git 操作。
