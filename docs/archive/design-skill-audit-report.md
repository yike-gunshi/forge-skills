> 审计日期：2026-03-26
> 审计范围：forge-design 现有能力 + 12 个外部设计 Skill 的完整内容
> 审计目标：诚实评估整合后的 forge-design 能否达到"充分设计后再开发，产出高质量交付"的目标

---

## 审计结论：7/10 — 有能力但有结构性短板

### 一句话判断

**整合后的知识库足够强大，但 forge-design 的流程设计有 3 个结构性问题会阻止惊艳设计的产出。**

知识不是瓶颈——99 条 UX 规则、161 个配色方案、反 AI-slop 清单、7 维度 0-10 评分、三层 Token 架构……这些外部 Skill 提供的设计知识已经远超人类设计师的日常参考范围。

**真正的瓶颈是：流程设计让 AI 没有机会把这些知识用好。**

---

## 三个结构性问题

### 问题一：forge-design 同时负责"设计"和"实现"，导致两者都做不好

**现状**：forge-design 的第5步是"实现设计变更"——修改 CSS、HTML、JS。

**问题**：
- 设计师和工程师是不同角色，混在一起会导致 AI 在"设计思维"和"工程思维"之间频繁切换
- 你的复盘记录证明了这一点：`2026-03-24-[前端UX精简]` 中，AI 在按钮样式设计上4轮返工，根因是"初始的设计决策全部错误"——不是实现问题，是设计决策不充分就进了实现
- design-consultation 和 plan-design-review **都不负责实现**——它们只做设计决策，实现交给后续流程
- Superpowers 也是这样分的：brainstorming → writing-plans → executing-plans，设计和实现严格分离

**建议**：
forge-design 只负责到第4步（DESIGN.md 产出），第5步"实现"交给 forge-eng。forge-design 的产出物是**足够详细的 DESIGN.md**，让 forge-eng 可以无歧义地实现。

**这意味着 DESIGN.md 的质量标准必须提高**——不是"大方向"，而是"具体到像素级的规范"，包含字体大小、间距值、颜色 Token、组件状态、响应式断点行为、动效参数。

### 问题二：第1步的设计探索太浅——4 个确认轮次不够

**现状**：从零创建 DESIGN.md 时，forge-design 用4轮确认：
1. 产品美学定位（"科技感/简约/温暖/专业？"）
2. 设计体系确认
3. 页面结构
4. 交互模式

**问题**：
- 第1轮问的是**标签**不是**方向**。"科技感"可以是深色终端风，也可以是苹果式极简白，也可以是赛博朋克霓虹——一个标签不够
- 缺少**竞品研究**——不知道同行怎么做的，就不知道什么是"基线"、什么是"差异化"
- 缺少**视觉参考**——文字描述的设计方向，用户和 AI 的理解可能完全不同
- design-consultation 做得好得多：
  - 先理解产品上下文
  - 用 WebSearch 搜索竞品网站
  - 用 browse 截图分析竞品设计语言
  - 提出 3+ 个完整美学方向，每个有 mood 描述
  - 区分 SAFE 选择（基线）和 RISK 选择（差异化）
  - 生成 HTML 预览页让用户直观看到效果

**建议**：
将 design-consultation 的 Phase 1-5 整合为 forge-design 第1步的子流程。用户在看到预览页、做过 SAFE/RISK 取舍之后，才进入第2步审计。

### 问题三：设计审计清单只有"是什么"，缺少"为什么"和"怎么修"

**现状**：80 项清单列出了检查项（如"行高：正文 1.4-1.6"），但没有：
- 为什么是这个值？（设计原理）
- 违反了怎么修？（具体指导）
- 这个项在不同产品类型中的优先级？（上下文敏感）

**对比**：
- ui-ux-pro-max 的 99 条 UX 规则每条都有**来源出处**（Apple HIG / Material Design / WCAG）
- plan-design-review 的 7 维度审查，每个维度评完分后**解释满分标准**然后**动手修**
- frontend-design 直接说"NEVER 用 Inter/Roboto/Arial"——有立场、有禁区

**建议**：
- 审计清单的每一类增加"为什么"栏（引用 Apple HIG / MD / WCAG 标准）
- 引入 ui-ux-pro-max 的规则作为"深度检查层"——80 项清单做快速扫描，发现问题后用 99 条规则做深度分析
- 引入 plan-design-review 的"评分 → 解释满分 → 修正"循环

## 对"惊艳设计"的诚实评估

### 能做到的

1. **设计知识不是瓶颈**。整合后的体系拥有：99条UX规则 + 80项审计清单 + 10个AI反面模式 + 字体黑名单 + 161配色方案 + 57字体搭配 + 三层Token架构 + 7维度0-10评分。这比绝大多数人类设计师的参考资料多得多。

2. **竞品调研有真实工具**。design-consultation 可以用 WebSearch 搜竞品、用 browse 截图分析——这不是"AI 凭记忆说竞品长什么样"，而是真的去看。

3. **反 AI-slop 意识很强**。frontend-design 的字体黑名单、design-consultation 的 AI slop anti-patterns、forge-design 的 10 个反面模式——三层防御。

4. **预览页可以直观确认**。在写代码之前就能看到字体+配色的实际渲染效果，减少"想象和实际不一样"的风险。

### 做不到的——需要清醒认识

1. **AI 没有真正的审美直觉**。所有这些规则和知识，本质上是"不犯错"的守则，不是"创造性突破"的能力。AI 可以做到 B+ 级设计（基础扎实、无明显问题、一致性好），但很难做到 A+ 级设计（让人"哇"的那种惊喜）。

2. **视觉细腻度有限**。AI 生成的 HTML 预览页和实际产品之间有很大差距——间距的微妙感、颜色在不同屏幕上的感觉、字体渲染的细节……这些需要人眼的最终判断。

3. **品牌感需要人类主导**。设计体系可以由 AI 提出方案，但"这个产品应该给人什么感觉"是一个需要产品创始人/设计主管拍板的主观判断。AI 的角色是**高效呈现选项**，不是**替代审美判断**。

4. **复杂交互设计仍需迭代**。拖拽、手势、复杂状态机……这些需要在实际使用中反复调整，设计文档覆盖不了所有 edge case。

### 现实预期

| 场景 | 整合前（现状） | 整合后（预期） | 天花板 |
|------|--------------|--------------|--------|
| 新项目从零开始 | D（直接写代码） | B+（充分设计后再写） | A-（需要人工微调） |
| 已有项目迭代 | C（设计粗糙直接改） | B（轻量审查后改） | A（需要完整重设计） |
| 反 AI-slop | C（10个模式检查） | A-（三层防御+黑名单） | A（外部Skill很强） |
| 设计一致性 | D（无检查机制） | B+（门控+一致性检查） | A-（需要人工审美判断） |
| 设计文档质量 | C（粗略描述） | A-（像素级规范） | A（Token+预览页+完整规范） |

## P0 必须解决的问题（按优先级）

1. **分离设计和实现**：forge-design 砍掉第5步，只到 DESIGN.md 产出为止
2. **引入竞品调研+美学探索**：整合 design-consultation Phase 1-5 到第1步
3. **引入分级门控**：L1/L2/L3 + 迭代不满触发器 + 子模块一致性
4. **引入 0-10 评分法**：替换或补充 A-F 等级，增加"满分标准"解释
5. **DESIGN.md 模板升级**：要求像素级规范，包含 Token、组件状态表、响应式行为
