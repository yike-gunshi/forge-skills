## 一句话摘要

从"一次只修一个 bug"升级到"**多窗口并行修不同功能域的 bug**"：靠项目根 `.forge/active.md` 心跳文件 + 业务视角的功能域判重 + 合并前 merge 预演 + 硬信号清理僵尸，解决"串行太慢、并行撞车"这对矛盾。

---

## 解决的用户痛点

forge-bugfix v5.0（上午发布）已经做到"一次一 bug + 双层验收清单 + 新发现必须分流"，但仍是**串行**——用户必须修完一个才能开下一个。真实场景里用户能同时照看 3–4 个 Claude Code 窗口，**瓶颈不在他，而在 AI 写代码的速度**。

> "如果我按照一个 bug 或一个功能的方式来修复，或者来新增，我的工作效率会特别低。所以我希望多开几个窗口，同时并行的推进 bug 的修复，或者是需求落地。"

直接多开窗口并行有三个硬问题：

1. **认知冲突**：两个窗口不知对方在修什么，AI 在 backlog 里可能推同一个 bug 给两个会话
2. **代码冲突**：等到 worktree 合并到 main 才发现两个改到同一个文件
3. **状态泄漏**：会话崩了、用户没走复盘，任务登记回收不掉，下次看 backlog 像一团乱麻

## 讨论过程的关键转折（值得记录的思考路径）

这一轮的价值不止在最终方案，而在**多次被用户纠正**之后才收敛到一个合理设计。记录下来给以后参考。

### 转折 1：先问澄清，不上来给方案

用户抛出需求后，AI 第一反应是列 5 个澄清问题——独立性标准、谁来分组、冲突兜底、需求迭代并行的层级、当前真实瓶颈。用户偏好里写过："脑爆时不要先建抽象，先问澄清"，这一条救了后面多次方向纠偏。

### 转折 2：AI 第一次提案"文件集判重" → 用户直接否决

AI 的第一个方案是"每个 bug 声明预计改动文件，判重看文件集交集"。这在技术上可验证、客观、容易自动化——但用户回复：

> "我觉得通过文件来判重是一个非常蠢的行为，不应该这么做。"

原因很具体：3 个 ASR 相关的 bug 可能横跨前端交互组件 + 后端 API，**文件集零重叠但上下文强相关**。文件集是技术视角，业务关联度为零。真正该用的是**功能域**——业务视角，能正确把"同一业务领域"的 bug 聚到一起。

> "我现在有三个 bug 都是和 ASR 有关的，可能是前端的设计，可能是后端的实现。那这种时候应该将它们放在一起，在一个会话中，通过不同的 worktree 去进行修复。"

**教训**：AI 倾向找客观可机器判定的指标，但这种指标不一定匹配人类的协作模式。业务视角往往比技术视角更符合真实场景。

### 转折 3：路由规则——同域归并 vs 异域并行

用户明确了使用场景：

- **同功能域的多个 bug** → 一个会话，多个 worktree，顺序修（上下文复用）
- **跨功能域的 bug** → 不同会话，真正并行

这比 AI 最初设想的"所有可并行的都并行"更精细。同域归并不是资源约束，而是**认知约束**——同一个会话里修完 ASR 的 bug A，理解会被带到 bug B，整体更快。

### 转折 4：session id 追溯

用户提议把 Claude Code 的 session id 写进 active.md，方便日后从 jsonl 文件回溯。AI 最初以为要用户手动粘贴（每次 P2 时问一句），用户反问"能不能自动拿"。

AI 去查 `~/.claude/` 目录，发现 `~/.claude/sessions/<pid>.json` 记录了每个进程的 sessionId，于是写了 `skills/forge-bugfix/scripts/get-session-id.sh`——通过 `$PPID` 向上回溯进程树，找到 Claude Code 进程的 pid，读对应 json。

**一次性解决**，以后 skill 在 P0 调它就能拿到 session id，不用用户手动输入。

### 转折 5：清理时机——用户否决时间戳

关于"会话崩了没清 active.md 怎么办"的兜底，AI 第一反应是加时间戳："超过 N 小时没更新算失联"。用户直接反对：

> "我希望你不要使用什么间隔时长，这种没有办法真正代表工作完成的方式来做标记。"

这是一句很有力的质问——**时间不代表工作完成**。用户可能离开 4 小时再回来继续修，AI 可能因为上下文长慢慢跑，"超过 N 小时"只会误杀活跃会话。

真正能代表"工作完成"的是**事实**：
- worktree 目录不存在了（用户手动删 = 真·弃用）
- 分支已 `merged` 到 main（工作已完成，只是漏清 = 真·完成）

这两个都是客观事实，不是启发式。v6.0 的 /forge-status 和 forge-fupan Phase 5.0 完全基于这两个信号。

### 转折 6：辅助 skill 的颗粒度

AI 最初提议拆两个 skill：`/forge-status`（只读巡检）+ `/forge-cleanup`（清理）。用户回复：

> "我建议 skill 合成一个比较好，简单一点"

合并后的 /forge-status 一条流程走完：扫 → 报告 → 问一次 y/n → 清理。Claude Code 里多一个命令就多一份记忆负担，合一能省就省。

### 转折 7：讨论完要先对齐再动手

方案聊得差不多时 AI 想直接动手，用户说：

> "我觉得我们需要梳理一下当前本次优化的背景、目标、我的反馈，以及我们最终希望达到效果，以及初步的方案，我要再做一次确认"

AI 写了一份完整"方案确认稿"，用户逐条 review、拍板最后几个遗留问题（功能域标签来源、重构型 bug 处理、辅助 skill 合一还是拆开），才进入实施阶段。

**教训**：大的跨 skill 改造别急着动手，先落成书面对齐，省掉后期返工。

## 最终方案（精炼版）

### 数据结构

**① 项目根 `.forge/active.md`（心跳文件，跨 worktree 共享）**

字段极简，四列：
```
- session: <sid> / worktree: <相对路径> / 任务: <BF-XX 或 FEAT-X> / 域: <功能域 或 域1,域2>
```

**明确不做的事**：不记 Phase、不记时间戳、不记优先级。active.md 只回答"谁在干啥"，不是任务管理器。

头部有"功能域声明"区：项目自己列功能域标签（登录、播放、识别 ...），AI 不得自创。首次使用时由用户编辑此清单再进 bugfix 流程。

**② `docs/bugfix/backlog.md` 加两列**

- `功能域`：和 active.md 保持一致
- `领取会话`：in-progress 时填 session id 前 12 位

### 路由规则

用户报新 bug → AI 推断功能域 + 你确认 → 读 active.md：
- **同域已有活跃会话** → 建议"切去那个会话顺序修"（上下文复用）
- **异域 / 无活跃会话** → 建议"新开窗口并行"
- **多域任务**（重构型）→ 任一域与已有会话冲突即判冲突

### 合并兜底

worktree 合并到 main 前强制 `git merge --no-commit --no-ff` 预演：
- 无冲突 → 正常合并
- 有冲突 → 回退 + 让用户决策（rebase / 暂存）

### 闭环

| 时刻 | 谁负责 | 动作 |
|---|---|---|
| 领取任务 | forge-bugfix P2/P3.1.5 | 写入 active.md |
| 正常结束 | forge-fupan Phase 5.0 | 按自己的 session id 清 active |
| 漏复盘兜底 | /forge-status | 扫硬信号，报告 → 用户确认 → 清理 |
| 合并成功 | forge-bugfix P7 | 只改 backlog 状态，**不动 active**（留给复盘/status） |

## 为什么这样设计（给读者看的 Why）

### 为什么文档驱动、而不是引入锁服务 / SQLite

用户场景是**单人多窗口**，不是多人真并发。加锁服务会过度工程化，维护成本高。文档 + 一个心跳文件够用，也符合 Forge 一贯的"文档驱动"哲学。

### 为什么 active.md 字段要极简

AI 越容易漏更新，整个机制就越脆弱。省掉"Phase / 时间戳 / 最后心跳"这些字段，AI 心智负担接近于零：只在"领取"和"释放"两个时刻写一次，中间不动。

### 为什么不依赖时间戳

时间不代表工作完成。worktree 存在性和分支合并状态是**事实**，是代码和文件系统本身的真相，不是推测。

### 为什么同域用同会话

**认知复用**比资源复用更有价值。同一个会话里修完 ASR bug A 后，对"ASR 数据流怎么跑"的理解直接带到 bug B，整体更快。如果拆成两个会话，每个都要重新读一遍代码。

### 为什么合并前要预演

并行场景下，两个会话可能都改到了同一个文件（即使功能域判重已经避开了大部分情况）。真正的代码冲突只能在 git merge 时暴露——预演一次是最便宜的兜底。

### 为什么 forge-status 和 cleanup 合一

多一个命令就多一份记忆负担。合一后的 /forge-status 一条流程走完（扫 → 报告 → y/n → 清理），不做就只报告不清理。这比拆两个 skill 简单。

## 反思：AI 在这轮里的过度设计

列一下被用户纠正的几次过度设计，作为以后的参考：

1. **文件集判重** → 被"业务视角不对"直接否决
2. **并行组预标签**（扫一遍 backlog 标 A/B/C 组）→ 被指出是多余的抽象，改成"每次领任务时实时算交集"
3. **时间戳兜底僵尸** → 被"时间不代表工作完成"直接否决
4. **两个辅助 skill**（status + cleanup 拆分） → 被"简单一点"合一
5. **Phase 状态同步**（想在 P3→P4→P5 每步都更新 active.md）→ 自己意识到太累赘，只在"领取"和"释放"两个时刻写

共同的教训：**AI 倾向于追求"周到"，结果是心智负担变重。真正好的机制是"自然对"——不需要每步都记得去维护，也不会出错**。
