# 2026-04-25 迭代：Bug 修复验收报告与 QA 自动闭环

**版本标签**：`v2026.04.25`
**主要 Skill**：`forge-bugfix`、`forge-qa`

---

## 一句话摘要

把单 bug 的“验收清单”升级为从发现到关闭的 **Bug 修复验收报告**，并定义 QA 自动闭环：forge-qa 发现 bug → 结构化登记 → forge-bugfix 独立 worktree/TDD 修复 → forge-qa 回归 → 最终一次人工验收。

---

## 背景

v5.0 的双层验收清单解决了“什么时候算修完”的问题，但真实使用里还有三个缺口：

1. 用户人工验收时看不到 QA 当时的逐步截图，只能读结论。
2. QA 使用的前后端地址、进程、worktree 是否和用户验收时一致，没有在报告里强展示。
3. 功能开发后的 QA 可能发现多个 bug，理想流程应该自动进入 bugfix 回修，而不是每个 bug 都打断用户。

---

## 核心改变

### 0. 封板原则

本次迭代把 bug 修复体系拆成五个稳定单元：

- **批次 = 管理单元**：负责聚合多个 bug 的状态、最终验收入口和跨会话进度。
- **Bug = 工程单元**：每个 bug 都有独立编号、独立报告、独立状态。
- **Worktree = 隔离单元**：每个 bug 在独立 worktree 中修复，避免互相污染。
- **TDD = 质量单元**：每个 bug 都要先有失败证明，再有修复后通过证明。
- **QA = 验收单元**：每个 bug 都由 forge-qa 回归，最终由用户或批次验收收口。

批量修复只改变管理方式，不改变单 bug 的工程边界。

### 1. 单 bug 文档改名与职责升级

路径仍兼容使用：

```text
docs/bugfix/reviews/BF-XXXX.md
```

但标题和语义升级为：

```markdown
# BF-XXXX Bug 修复验收报告
```

这份报告从 bug 被登记或领取时就创建，不再等修完后才生成。

### 2. 报告内容升级

报告固定包含：

- 当前状态
- 问题发现记录
- 复现记录
- 根因分析
- 修复记录
- TDD / 回归用例
- 验收入口与环境身份校验
- 人工验收指南
- QA 测试过程与截图证据
- QA 自动闭环记录
- 最终结论与新发现分流

截图保存在：

```text
docs/bugfix/reviews/assets/BF-XXXX/
```

并用 Markdown 直接内嵌：

```markdown
![](./assets/BF-XXXX/qa-1-01-open-page.png)
```

### 3. 环境一致性变成硬门禁

forge-qa 必须在报告中展示并校验：

- Frontend URL
- Backend/API URL
- URL 来源
- PID
- cwd
- branch/commit
- health check 或关键接口探活

QA 用的地址必须和用户验收地址一致。能拿到 PID/cwd 时，cwd 必须指向当前 worktree；否则 QA_FAIL。

### 3.5 Codex Browser Use 作为前端验收表面

在 Codex 环境中，前端页面/交互/视觉/控制台相关 QA 优先使用 `browser-use:browser`：

- 打开本地 `APP_URL`
- 通过 DOM snapshot 定位元素，不盲猜选择器
- 点击、输入、切换、提交后采集关键状态截图
- 截图写入 Bug 修复验收报告并 Markdown 内嵌
- console error 进入 FAIL

Playwright 仍负责可重复脚本断言；gstack/browse 可作为已有项目补充；Computer Use 只在 Browser Use 不可用或目标不是浏览器页面时兜底，并必须在报告里写明原因。

### 4. 批量只是编排，单 bug 仍是工程单元

无论是一个会话里快速闭环，还是多天、多会话处理大量 bug，底层规则不变：

- 每个 bug 独立 BF 报告
- 每个 bug 独立 worktree
- 每个 bug 独立 TDD
- 每个 bug 独立 commit
- 每个 bug 独立 QA 回归

批量流程只能汇总状态，不能把多个 bug 合成一个修复单元。

### 5. QA 自动闭环有上限

自动闭环不是无限循环。

以下情况必须升级用户：

- 同一个 bug 连续回修失败 3 次
- 问题可能是新需求而不是 bug
- 修复方案明显扩大改动面
- 视觉/交互预期无法用 Feature Spec 或 DESIGN.md 判断
- 前后端环境身份无法确认一致
- 多个 bug 互相牵连，无法判断优先级

升级时 AI 必须给出决策卡：问题、证据、已尝试方案、当前疑问、可选决策和推荐理由。

## 新增状态

`docs/bugfix/backlog.md` 支持以下状态：

| 状态 | 含义 |
|---|---|
| `pending` | 待修 |
| `in-progress` | 正在独立 worktree 修 |
| `fixed-awaiting-qa` | 代码已修并提交，等待 forge-qa 回归 |
| `qa-failed` | forge-qa 回归失败，需回到 bugfix |
| `qa-pass-pending-final-review` | 自动 QA 已通过，等待最终人工验收 |
| `blocked-human` | 自动闭环遇到疑问，等待用户判断 |
| `deferred` | 确认延期，不进入当前批次 |
| `resolved` | 已修复并通过最终验收 |
| `wontfix` | 决定不修 |

## 两种使用模式

### 模式 A：功能开发后的 QA 自动闭环

```text
功能开发完成
→ forge-qa 全量/差异 QA
→ 发现 bug 并生成结构化 BF 候选
→ forge-bugfix 逐个独立修复
→ forge-qa 逐个回归
→ 全部 QA 绿
→ 生成最终人工验收包
→ 用户统一验收一次
```

### 模式 B：多天/多会话 BugFix 批量修复

```text
直接从 backlog 或 QA 报告进入
→ 多个会话按功能域领取不同 bug
→ 每个 bug 独立 worktree/TDD/QA
→ 状态汇总到 backlog 和批次报告
→ 最后统一人工验收
```

可选批次汇总路径：

```text
docs/bugfix/batches/BATCH-YYYY-MM-DD.md
```

批次汇总只做最终验收入口，不替代单 bug 报告。

## 设计决策

### 为什么不改路径

继续使用 `docs/bugfix/reviews/`，兼容历史 backlog 链接、`review_doc` 参数和已有 Mode B 逻辑。语义通过文档标题和模板升级完成。

### 为什么不让 forge-qa 自己修 bug

forge-qa 只测不修，forge-bugfix 只修单 bug。自动一条龙由调度层编排。这样既能减少用户介入，也不会让测试过程顺手改代码。

### 为什么每个 bug 仍然独立 worktree

批量的目标是减少人工打断，不是降低隔离标准。独立 worktree/TDD/commit/QA 是质量单元，批次只是管理单元。
