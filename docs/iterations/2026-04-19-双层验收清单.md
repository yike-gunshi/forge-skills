# 2026-04-19 迭代：Bug 修复双层验收清单

**版本标签**：`v2026.04.19`
**主要 Skill**：`forge-bugfix`（主要改造）、`forge-qa`（新增 Mode B）
**CLAUDE.md 规则**：新增 R11 + 沟通规则

---

## 一句话摘要

给 bug 修复加上**结构化双层验收清单**（forge-qa 自动 + 用户人工），解决"修到第 5 个还没封闭"的边界模糊问题；新发现一律分流到 backlog，禁止当场顺手修。

---

## 解决的用户痛点

过去 forge-bugfix 已经做到"一次一 bug + 独立 worktree"，但仍存在：

1. **封闭边界模糊**：AI 跑完自动 QA 后用户"模糊确认"，没有明确的 Pass/Fail 节点 → 修复链不断延伸
2. **新发现污染当前修复**：验收过程中用户反馈的新问题，要么被 AI 当场顺手修（scope 蔓延），要么被塞进单个 TODO 文件混在一起
3. **QA 散乱**：每次修复产出截图、日志、测试输出散落各处，没有"这个 bug 验收的真相"集中地

---

## 核心改变（5 点）

### 1. 每次修复生成表格化验收清单文档

`docs/bugfix/reviews/BF-{MMDD}-{N}.md`，单 bug 单文档。包含：

- 基本信息（Bug 描述、根因、修复做了什么、涉及文件、commit）
- 验证清单表格（验证项 / 预期效果 / 操作步骤 / QA 验证 / 你的验收）
- 最终结论三选一：Pass / Fail / Pass + 新发现
- 新发现区

### 2. 双层验收

- **第一层**：`forge-qa` 的新 Mode B 模式自动跑测试，回填清单的 "QA 验证" 列
- **第二层**：用户打开文档做人工验收，填 "你的验收" 列和最终结论
- 用户填完最终结论、说 "验收了"，AI 才进入合并

### 3. 新发现强制分流

新发现一律写进 `docs/bugfix/backlog.md` 的对应区：

- 🐛 **待修 bug**（独立，分配 BF-XX 编号 + 优先级 P0/P1/P2）
- 💡 **新需求**（建议走 `/forge-prd` 立项）
- 🌀 **待澄清反馈**（下次复现时再处理）
- 🗄️ **已处理**（永久保留，历史回溯用）

**同根判定必须举证**（同文件 / 同函数 / 同数据流），举证不足默认独立 bug。

### 4. `forge-qa` 新增 Mode B

保留原有 Mode A（完整 QA）不变。Mode B 专门服务 `forge-bugfix` 的 P6 调用：
- 通过 Skill 工具调用，强制传入 `review_doc` 路径 + `bug_id` + `worktree` 等参数
- 只测不修，结果回填清单，不再产出散乱的 QA 报告

### 5. 默认建议新会话

下一个 bug 的默认建议从"回 P2 启动下一轮"改成 **"新会话或 /clear、/compact"**，避免长会话的 scope 蔓延。

## 新流程图速览

```
P5 修复完成
  ↓ 生成 docs/bugfix/reviews/BF-XX.md
P6 Skill(skill="forge-qa", args="mode=B review_doc=... bug_id=...")
  ↓ QA 回填清单"QA 验证"列
  ├─ 全过 → P6.5
  └─ 有挂 → 回 P5
P6.5 用户打开清单，填"你的验收"列 + 最终结论，说"验收了"
  ↓ AI 读清单
P7 分流：
  ├─ Pass → 合并 worktree
  ├─ Fail → 回 P5（不接新问题）
  └─ Pass + 新发现 → 合并 + P7.5
P7.5 新发现分类写进 backlog.md
  ↓ 同根要举证 / 独立 → 🐛 / 新需求 → 💡 / 模糊 → 🌀
P8 归档 backlog + 写 bugfix 索引
  ↓
出口：默认建议新会话或 /clear /compact
```

## 设计决策的 why（给以后读这个文档的你）

### 为什么不做"统一 FEEDBACK.md 流水账"

讨论过程中一度想做"AI 无损记录所有反馈到 FEEDBACK.md，分类用户每周整理"——被对抗性审查和用户本人推翻。真问题不是"记哪里"，是"修复没有明确封闭边界 + 反馈没有行动化出口"。流水账只是把问题往后拖。

### 为什么抛弃"主线映射"方向

第 1 轮讨论 AI 想做"主线-skill 关联 + 主线切换维护"，用户直接否定："主线是我心里的事，不应由你维护"。保留了正确的"用户主动声明、AI 不猜"的节奏，聚焦到真痛点：**解耦**。

### 为什么同根判定要举证

历史上 AI 喜欢把"看起来相关"的 bug 声称共因一起修，结果改动面扩大。要求 AI 拿出"同文件/同函数/同数据流"的可验证证据，才能并入当前修复——降低 scope 蔓延概率。

### 为什么默认新会话而不是 /clear

两种都可以，让用户选。但默认推荐新会话——避免"同会话上下文虽 clear 但 AI 习惯性记得之前 bug 的细节"（实际观察过）。
