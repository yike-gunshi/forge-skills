## 一、用户在讨论什么（本轮最新定义）

一个**贯穿 forge 体系的反馈捕捉机制**。目的：

- 用户随时反馈**不被打断**
- AI 按**某条主线**顺序处理，不来回拉扯
- 用户能看到**真实记录**、**真实进度**
- **精准可控**

---

## 二、用户明确的两个反馈场景

### 场景 1：bugfix / QA 过程中的反馈

**触发时机**：正在验收 bug、bugfix 中、QA 中

**反馈内容**：人为验证结果 + 发现的新问题

**反馈与当前 bug 的 3 种关系**：
1. **和当前 bug 有关** → AI 应该一并修
2. **是当前 bug 衍生出来的** → AI 应该一并修
3. **和当前 bug 无关** → AI 记下来，下次再修

**下次修复时的取单规则**：
- AI 找**最高优先级**的，或者**和当前 bug 最相关**的先修

**用户视角期望**：
1. 没被打断
2. 顺序推进，不来回拉扯
3. 能看到真实记录
4. 能看到真实进度

### 场景 2：会话不活跃时的反馈

**触发时机**：用户在使用产品时自己发现了某个问题（AI 此时可能根本没在做事）

**反馈内容**：新 bug 或新需求

**直接处理**：用户打开对话直接说出来

## 三、用户对机制形态的思考

- 希望有一个**专门的捕捉机制**
- 初步想法：关键词 / 专门的 skill 做触发（最标准、AI 一定识别得到）
- 触发后：AI 识别当前所处阶段（比如 bugfix 的 P5），给出对应处理方案
- 疑问：**是否应该做一个贯穿 forge 体系的 skill（比如叫 `/forge-feedback`）？**
- 顾虑：bugfix 会话中临时加载这个 skill 会**浪费大量上下文**
  - 那种场景下用户倾向于**直接说**，不加载 skill
- 结论：做成独立 skill 还是别的形态，**还没定**

## 七、待论证的关键问题（按优先级排序）

这些问题没讨论清楚，**不出方案**：

### Q1（最根本）：什么叫"一条主线"？

用户说"一直是沿着某一条主线在做"——这是所有机制的根基。主线是什么？

可能的候选：
- **a)** 一个 feature（如"AI 摘要"、"B站抓取"）
- **b)** 一个用户场景（如"订阅管理流程"）
- **c)** 一个版本（如 v12.3）
- **d)** 当前手头这个 bug 为圆心的一组相关 bug
- **e)** 用户每天决定的"今天的主线"（用户自己声明）

主线怎么记录？怎么切换？切换时反馈池里的东西怎么流转？

### Q2：场景 1 里"相关 / 衍生 / 无关"由谁判断？

- 用户当场标？（把负担给你）
- AI 当场判断？（容易错，forge-bugfix P3.4 有"共因判断"但只限技术视角）
- 混合：AI 先判，用户覆盖？

判断错了的代价：
- 错判"相关" → 当前 bug 膨胀成小重构，违反单 bug 原则
- 错判"无关" → 漏掉真正的共因，修完没几天又复发

### Q3：下次修复的"优先级"和"相关性"怎么算？

用户说"下次找最高优先级的，或最相关的"——需要算法：
- 优先级：严重度？影响用户数？你主观标 P0/P1？新旧？
- 相关性：同一 feature？同一文件？同一模块？
- 谁排序？AI 自动 + 你覆盖？

### Q4：捕捉机制的触发器，到底是什么

三个候选：
- **a)** 用户专门打某个命令（如 `/fb 内容` 或 `/feedback 内容`）
- **b)** 启动专门 skill（如 `/forge-feedback`），skill 承担上下文感知、写文档、提供建议
- **c)** 让 CLAUDE.md 规则"在任何 skill 里自动捕捉用户反馈"

用户担心 (b) 浪费上下文。(a) 最轻但是用户需要记命令。(c) 最隐式但依赖 AI 自觉。

### Q5：上下文节约具体多严重

用户提到"加载 skill 浪费上下文"。要量化：
- 一个 skill 的 SKILL.md 是多大？（forge-bugfix 大约 3.2k 行）
- 一次加载会占用多少 token？
- 反馈 capture 的 skill 真的需要多大？会不会只需要 100 行？

如果 capture skill 本身只有 100 行，加载成本几乎可忽略。这个顾虑可能被高估了。

### Q6：场景 2 里 AI 怎么"识别当前阶段"

如果用户在一个陈旧的 bugfix 会话中反馈新问题，AI 要知道当前在 P5 还是 P7。
- 靠 TodoWrite 的状态？
- 靠 skill 里维护的某个变量？
- 靠 git log / worktree list 推断？

### Q7：一次 dump 多条反馈怎么办

真实使用中你可能一次性讲 5 条反馈（因为刚刚试用完一批）。捕捉机制要支持批量吗？还是每条单独？

### Q1 讨论（2026-04-19 完成）

**用户澄清**：

- 主线 = **"现在开始做什么"** 的显式声明
- 粒度：一个功能 / 一个模块 / 一类任务
- 例子：
  - "视频的 ASR 能力"（功能粒度）
  - "历史 bug 的修复"（一类任务粒度，可能下挂多个具体 bug）
- 主线**由用户手动声明**，AI 不得自动推断
- 主线**会频繁切换**
- 一个会话**大概率**维护一条主线（不绝对）
- 会话不等于主线（一条主线可能跨多个会话）
- 会话归档后，下次新启会话时 **AI 必须主动问：这次接哪条主线 / 新开哪条主线**
- **关键交互偏好**：主线切换、续接是"AI 问、用户答"的模式，不是"AI 判断、用户纠正"

**Q1 结论**：

> 主线是一个"用户声明→生命周期→终止或归档"的独立实体。
> 任何反馈捕捉机制都必须能**绑定主线**，否则"沿着主线修下一个"这句话落不了地。

### 本轮已达成的共识

- ✅ 每次新会话开始，AI 必须主动问"这次的主线是什么"
- ✅ 主线由用户手动声明，AI 不自动推断
- ✅ 主线的命名粒度：功能 / 模块 / 一类任务
- ✅ 主线会在不同会话间切换和接续
- ✅ 反馈记录必须能关联主线（具体怎么关联，Q2 讨论）
- ✅ 交互默认模式：AI 问、用户答（不是 AI 自作主张判断）

### Q2 讨论（2026-04-19 完成）

**用户澄清（非常重要）**：

- **第 1 点**：反馈和主线的匹配是 **AI 的责任**，用户只管 dump 内容
  - AI 负责整理、分类、主线重启时捞出来
  - 机制兜底，用户忘了也没关系，会话中问 AI 或 AI 主动告知
- **第 2 点**：表面上跨主线、实际同根的问题，AI **必须主动识别**
  - 告诉用户、主动分类和优化
  - AI 在主线归属上要**多做一步**
- **第 3 点**（推翻我之前的假设）：**Bug 修复主线一定限定在某个模块，做完就结束**
  - 之前我以为"bug 修复"是永久 active 的类别主线 → 错
  - 实际：bug 修复主线和功能开发主线一样，都是**限定范围的一次推进**
  - 所以"轮（round）"的概念不需要，直接主线结束/归档就行

**Q2 结论**：

> **主线 = 限定范围的一次工作推进，有明确开始和结束。**
>
> 不分"类型 × 生命周期"两套模型。统一为：
> - 主线声明时自带范围（"视频 ASR 能力" / "订阅模块的历史 bug"）
> - 做完归档
> - 下次类似的开新主线（不复用旧主线）
>
> 反馈的归属是 AI 的活：记录 + 分拣 + 跨主线识别同根 + 主线重启时捞回。

### 更新后的共识清单

- ✅ 每次新会话开始，AI 主动问"这次的主线是什么"
- ✅ 主线由用户手动声明，AI 不自动推断
- ✅ 主线命名自带范围（功能名 / 模块+工作类型）
- ✅ 主线有明确终点，做完归档
- ✅ 反馈的分拣、归类、同根识别、重启捞回 → **都是 AI 的责任，用户只管说**
- ✅ 交互默认模式：AI 问、用户答，不是 AI 猜

### Q3 讨论（被用户推翻 → 转向）

**用户反馈**：主线映射方式很愚蠢，主线是用户心里的事，不需要 AI 维护一个文档。

**真正的核心痛点（用户明确说最痛的）**：
1. 每次工作没法解耦——修 bug 时冒出的新东西污染当前工作
2. **bug 修复没有明确的"完工"边界**——不知道到哪一步算修完、算验收完
3. 验收发现新 bug 时，不知道应该继续本会话还是新开
4. 反馈离开上下文就失去意义，需要**转化为可执行动作**，不是流水账记录

**转向后的设计方向**：
- **不维护主线**——用户自己心里有主线
- **反馈动作化**——每条反馈当场分类转成可执行条目或合并到当前修复
- **bug 修复必须有明确封闭节点**（User Gate）
- **默认新 bug 新开会话**（用户偏好：新会话 或 clear/compact 后继续）

### Q4 后续讨论（2026-04-19 继续）

**用户补充要求**：

1. **一个 bug 一个文档**：AI 生成后在会话中给路径，用户点击跳转
2. **表格式清单**：详细到每个验证项，字段包括"改了什么、预期效果、操作步骤、QA 验证、用户验证"
3. **双层验收**：
   - 每次修完先让 **forge-qa 做一轮自动验收**（填 QA 那一列）
   - QA 过了再交给用户做 **二次人工验收**（填用户那一列）
   - 验收清单**同时给 QA 和用户看**，double check

**核心设计已定**：

| 点 | 结论 |
|---|---|
| 验收清单形态 | 一个 bug 一个 markdown 文档，表格为主 |
| 存放位置 | `docs/bugfix/reviews/BF-{日期}-{N}.md` |
| 表格字段 | Bug 描述、根因、改了啥、涉及文件 + 验证项表格（验证项、预期、步骤、QA 验证、用户验收）|
| 双层验收 | forge-qa 先跑 → 全过 → 交用户二次验收 |
| 最终结论 | Pass / Fail / Pass + 新发现（三选一） |
| 新发现去哪 | 独立 bug → `docs/bugfix/backlog.md`；新需求 → 建议 forge-prd |
| AI 通知方式 | 会话中给文档路径，用户点击跳转 |
| 用户推进信号 | 用户说"验收了"，AI 读文档 |

**流程图（最终版）**：

```
P5 修复完成
  ↓
AI 生成验收清单 docs/bugfix/reviews/BF-XX.md（填"改了啥+验证项"）
  ↓
P6 forge-qa 接手
  └─ 在清单里填"QA 验证"那一列
  ↓
  ├─ QA 全过 → 告诉用户"请人工验收 @ 文档路径"
  └─ QA 有挂 → 回 P5 继续修，不打扰用户
  ↓
用户打开文档，填"用户验证"列 + 最终结论
  ↓
用户说"验收了" → AI 读文档
  ├─ ✅ Pass → P7 合并
  ├─ ❌ Fail → 回 P5
  └─ ⚠️ Pass + 新发现 → P7 合并（本次封闭）+ 把新发现分流到 backlog
```
