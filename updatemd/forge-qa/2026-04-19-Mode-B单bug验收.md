# forge-qa：新增 Mode B 单 bug 验收清单模式

**日期**：2026-04-19
**来源**：配合 `forge-bugfix` 双层验收清单改造，需要一个专门服务单 bug 验收的 QA 模式

---

## 优化背景

原 `forge-qa` 是面向一次完整 QA 流程设计的（test-spec 生成 → 10 维度断言引擎 → QA.md 报告 → User Gate）。`forge-bugfix` 改造要"每次修完调 forge-qa"，但直接跑完整 Mode A 显然过重：
- 单 bug 不需要 test-spec 生成（验收项已经在清单里）
- 单 bug 不需要 User Gate（那一步挪到 forge-bugfix 的 P6.5）
- 单 bug 需要把结果回填到清单，而不是生成独立的 QA.md

因此增加 Mode B，和 Mode A 并存，通过调用参数区分。

---

## 方案设计

### 模式判断（优先级从高到低）

| 优先级 | 触发条件 | 模式 |
|---|---|---|
| 1 | Skill 调用 args 含 `mode=B` 和 `review_doc=<路径>` | B |
| 2 | 触发消息含 `forge-bugfix` / `review-checklist` / `docs/bugfix/reviews/BF-\d+-\d+.md` | B（启发式）|
| 3 | 其他情况 | A |

### Mode B 入参契约（forge-bugfix 必须传）

| 参数 | 必填 | 含义 |
|---|---|---|
| `mode=B` | ✅ | 强制信号 |
| `review_doc=<路径>` | ✅ | 验收清单文档（不存在直接 exit 1）|
| `bug_id=BF-{MMDD}-{N}` | ✅ | 用于命名截图 / 日志 |
| `worktree=<路径>` | ✅ | 在该 worktree 内运行测试 |
| `commit=<hash>` | ✅ | 修复范围定位 |
| `app_url=<URL>` | 条件 | 仅 bug 类型涉及应用运行时 |

### Mode B 行为

1. 读取 `review_doc` 清单的基本信息 + 验证项表格
2. 为每个验证项选择测试引擎（UI/视觉 → gstack 或 Playwright；API/逻辑 → curl 或单元测试；响应式/可访问性 → Playwright；静态属性 → Grep/Bash）
3. 执行验证 + 回填清单的"QA 验证"列：✅ 通过 / ❌ 未通过（附证据链接）
4. 填充"QA 验收记录"节（跑了哪些测试、每条 pass/fail 详情、附截图）
5. 控制台零容忍（page error / console.error 自动 FAIL）
6. 进程身份核对（`ps aux | grep <服务>` + `lsof -p $PID | grep cwd`）
7. 状态信号返回给 forge-bugfix：
   - `QA_PASS`：所有验证项 ✅ → forge-bugfix 进 P6.5
   - `QA_FAIL`：至少一条 ❌ → forge-bugfix 回 P5

### Mode B 不做的事

- ❌ 不生成 `docs/QA.md`（那是 Mode A 的产物）
- ❌ 不做 User Gate（由 forge-bugfix 的 P6.5 做）
- ❌ 不写 FEEDBACK.md（那是 Mode A 的 reject 闭环）
- ❌ 不主动判断"修复是否正确"（由用户在 P6.5 决定）
- ❌ 不修改代码（和 Mode A 一样，只测不修）

## 改动清单

### SKILL.md

- **frontmatter** description 扩展说明两种模式
- **顶部新增"调用模式"章节**：模式判断脚本 + 入参契约表
- **文末新增"Mode B：单 bug 验收清单模式"章节**（B.1 到 B.9 九个子节）
- **清理版本号**：`v3.1` / "v3.1 新增" 等标注清掉（skill 只写当前能力）

### Mode A 不变

原有的 10 维度 Playwright 断言引擎、test-spec 生成、三层架构、完整流程全部保留不变。

## 回测

以本次讨论的输入输出契约为基准：

| 场景 | Mode B 表现 |
|---|---|
| forge-bugfix P6 显式调用 `Skill(skill="forge-qa", args="mode=B ...")` | ✅ 优先级 1 命中 |
| 用户直接 `/forge-qa`（不带 review_doc） | ✅ 走 Mode A（默认） |
| 消息里提到 `docs/bugfix/reviews/BF-0419-2.md` 但没显式 mode 参数 | ✅ 优先级 2 启发式识别为 B |
| review_doc 路径不存在 | ✅ 必需参数校验 `exit 1`，不静默继续 |
