# CLAUDE.md：新增 R11（双层验收清单 + 新发现行动化）

**日期**：2026-04-19
**来源**：`forge-bugfix` 双层验收清单改造 / 用户痛点反馈"修到第 5 个还没封闭"

---

## 优化背景

`forge-bugfix` v4.0 已经做到"一次一 bug + worktree 隔离"，但验收阶段仍是 AI 隐式判断 + 用户模糊回复，导致：

1. 封闭边界模糊：用户不知道什么时候算"修完"
2. 新发现污染当前修复：验收时反馈的新问题要么被顺手修（scope 蔓延），要么塞进 TODO 散落
3. QA 散乱：每次修复的证据缺乏集中地

R11 规则在 CLAUDE.md 层面强制要求双层验收清单机制。

---

## R11 规则全文

> 以下是模板。其他用户可抄到自己的 `CLAUDE.md` 的"规范驱动规则"节。

```markdown
#### R11: Bug 修复走双层验收清单 + 新发现必须分流

- **每次 bug 修复 SHALL 生成结构化验收清单文档** `docs/bugfix/reviews/BF-{MMDD}-{N}.md`
  - AI 填基本信息 + 验证项前 3 列（验证项 / 预期效果 / 操作步骤）
  - 验证项至少 3 条，其中至少 1 条边界/异常态
  - AI 不得填 "QA 验证" 和 "你的验收" 两列
- **必须先由 forge-qa 做自动验收**（填 "QA 验证" 列），**QA 全过后**才让用户做人工二次验收（填 "你的验收" 列 + 最终结论）
- **没有用户最终结论，不得进入合并步骤**（即使 QA 全过）
- **用户验收结果是三选一**：Pass / Fail / Pass + 新发现
  - Pass → 合并 worktree
  - Fail → 回 P5 继续修（不接受新问题进来）
  - Pass + 新发现 → 先合并当前修复 → 分流新发现
- **新发现必须写入 `docs/bugfix/backlog.md` 对应区**，禁止当场顺手修：
  - 独立 bug → 🐛 待修区（分配 BF-XX 编号 + 优先级 P0/P1/P2）
  - 新需求 → 💡 新需求区（建议走 /forge-prd 立项）
  - 模糊反馈 → 🌀 待澄清区
  - 同根（并入当前修复）→ **AI 必须举证**（同文件的同一函数 / 同一数据字段 / 同一上游依赖），举证不足默认独立 bug
- **backlog.md 的 🗄️ 已处理区永久保留**，不清理（用于以后定位问题时回溯）
- **AI 自己想到的顺手重构 SHALL NOT 写进 backlog**，写代码文件里的 `// REFACTOR: {想法}` 注释即可
- **下一个 bug 默认建议新会话或 /clear、/compact**，避免长会话的 scope 蔓延
- 具体流程见 `skills/forge-bugfix/SKILL.md` 的 P5.3 / P6 / P6.5 / P7 / P7.5
```

## 配套：文档位置小节追加

```markdown
- **Bug 任务池**：`docs/bugfix/backlog.md`（forge-bugfix 单一入口）
- **Bug 验收清单**：`docs/bugfix/reviews/BF-{MMDD}-{N}.md`（每个 bug 一个文档）
- Bug 历史汇总（可选）：`docs/bugfix/{日期}.md`
```

## 配套：沟通规则（单独一节）

同一天新增了 CLAUDE.md 的"沟通规则"节，要求 AI 和用户交流时不用内部术语（弹道 / SSOT / Scope Guard / User Gate / Capture box / 验收巡场 / rolling triage / MVF / bumble bee），每个禁用词都给中文替代说法。这一节的内容是项目级偏好，不做通用模板，按需自定义。

## 回测

以本次讨论暴露的 3 个原始痛点为基准：

| 痛点 | R11 是否拦得住 |
|---|---|
| "修到第 5 个还没封闭" | ✅ 用户最终结论是硬性封闭节点 |
| 新发现顺手修导致 scope 蔓延 | ✅ 硬性分流 + 同根必须举证 |
| QA 证据散乱 | ✅ 集中到单份验收清单文档 |
