# forge-bugfix v2.1 → v3.0：跨会话知识 + 需求分流 + 经验沉淀 + 方案确认

**日期**: 2026-04-18
**来源**: 54 篇复盘文档交叉分析 + 业界调研 + 用户反馈（3 个实际痛点）

---

## 一、优化背景

### 用户反馈的 3 个真实痛点

1. **跨会话 bug 知识断裂**：今天修了一个 bug，明天新会话又重新描述、重新排查。bugfix 文档已建立，但 skill 的 P1 步骤不读它。
2. **修 bug 时夹带新需求**：修复过程中发现"这里应该加个功能"，然后 bug 修复和新功能搅在一起，越搞越乱。
3. **bugfix skill 不进化**：每次修 bug 的经验没有沉淀回 memory 或 skill 规则，同类问题下次换会话还是从零开始。

### 业界调研发现的 2 个差距

1. **缺少修复方案确认环节**（来源：mcpmarket.com Structured Bugfix Workflow）：AI 找到根因后直接开始改代码，没有给用户解释根因+确认方案的环节。
2. **三振出局缺少"清上下文"选项**（来源：Cursor Agent Debug Mode 最佳实践）：长对话中上下文噪声会导致 AI 判断力下降，连续失败时应该提供"清空重来"的选项。

### 业界调研的其他发现（本次未采纳，记录备查）

- **DAP 断点调试**（AlmogBaku/debug-skill）：通过 Debug Adapter Protocol 让 AI 设断点、单步执行。当前依赖临时日志验证假设，DAP 对复杂状态 bug 更高效。→ 需要安装依赖，暂不引入。
- **CI 自动触发轻量版**（OpenAI Codex Autofix CI）：测试失败自动触发 bugfix，成功自动开 PR。→ 需要 CI 集成，暂不引入。
- **因果树结构化排除**（FVDebug 学术研究）：多假设时先构建因果树再排除，替代线性逐个试。→ 对用户透明，可后续内部优化。
- **defense-in-depth**（obra/superpowers）：修复时不只修根因，同时在上下游加守卫断言。→ 已有 P5.5 同类模式扫描覆盖类似需求。

## 二、问题诊断

### 痛点 1：跨会话知识断裂

**根因**：forge-bugfix P1 只读 PRD + ENGINEERING.md，不读 `docs/bugfix/*.md` 和 Memory。每个新会话是"失忆"状态。

### 痛点 2：需求混入 bug 修复

**根因**：forge-bugfix 没有「需求隔离」机制。P4 修复阶段没有检查用户描述的内容是否超出 bug 修复范围。

### 痛点 3：经验不沉淀

**根因**：P5 只产出当次报告，不回流到 bugfix 文档、Memory 或 skill 规则。

### 调研差距 1：方案不确认

**根因**：P3 假设验证通过后直接跳到 P4 实现。同一 bug 可能有多种修法，AI 自选一种可能不是用户想要的。用户也无法从中学习到 bug 的原因。

### 调研差距 2：三振后无法重启

**根因**：长对话中积累的错误假设和噪声会干扰 AI 判断，但三振出局只提供"继续/升级/埋点"三个选项，没有"重新来过"。

## 三、方案设计

### 改动 1：P0 + P1 — 读取 bugfix 历史 + Memory

- P0 前置脚本新增 `$DOC_BUGFIX` 变量，检测 `docs/bugfix/` 目录
- P1 文档表新增两行：Bugfix 历史（必须）、Memory feedback 条目（必须）
- P1 新增「Bugfix 历史检索」子流程：用关键词搜索历史 bug 记录，匹配到时询问用户是否沿用历史根因

### 改动 2：P3.4 — 根因解释 + 修复方案确认

- 在 P3（假设验证）和 P4（实现修复）之间新增 P3.4 步骤
- 用通俗语言（非代码术语）向用户解释根因的因果关系
- 列出 A/B 修复方案，推荐其一，用户确认后才进入 P4
- 设计原则：**这既是质量门控，也是用户的知识成长机会**

### 改动 3：P4.0 — 需求分流（Scope Guard）

- 在 P4 修复阶段最前面新增 P4.0 步骤
- 识别信号：用户说"顺便加一个"、验收反馈超出 PRD 范围
- 分流动作：记录到 bugfix 文档的「衍生需求」区 + TODOS.md
- 默认推荐"先记下来，修完 bug 再说"

### 改动 4：P5.7 — 经验沉淀

- 在 P5.5（同类模式扫描）之后新增 P5.7 步骤
- 写入 bugfix 文档（标准化格式，含经验教训字段）
- 判断是否值得写入 Memory（跨项目/跨会话有价值的模式才写）
- 更新历史 bugfix 条目的交叉引用（如果是同一问题复发）

### 改动 5：P3.3 — 三振出局新增"清空重来"选项

- 原有 A/B/C 选项不变，新增 D) 清空重来
- AI 总结已排除的假设和已收集的证据，建议用户开新会话粘贴摘要

### 改动 6：规则更新

- 规则 2 扩展：强制读 PRD + ENGINEERING + **Bugfix 历史**
- 新增规则 3：根因必须用通俗语言向用户解释，用户确认后才动手
- 新增规则 12：新需求走分流，不混入 bug 修复
- 新增规则 13：修完就沉淀，写入 bugfix 文档 + Memory

## 五、用户体验变化

用户在使用 forge-bugfix 时的具体变化：

| 场景 | 之前 | 之后 |
|------|------|------|
| 报告一个之前修过的 bug | AI 从零排查，浪费 30 分钟 | AI 说"这和 BF-0417-3 类似，上次根因是 XXX，要沿用吗？" |
| AI 找到根因后 | 直接开始改代码 | 先用通俗语言解释原因，列出修复方案让你选 |
| 修 bug 时想到新功能 | 混在一起做，越搞越乱 | AI 说"这是新需求，我先记到 TODO，修完 bug 再说" |
| 修完 bug 后 | 报告完就结束 | 自动写入 bugfix 文档 + 有价值的经验写入 Memory |
| AI 连续 3 次猜错 | 只能"继续/升级/埋点" | 多一个"清空重来"选项 |
