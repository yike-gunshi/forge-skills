## 二、v3.1 新增改动（5 项）

### 改动 1：P1.5 — 多 Bug 分诊（Multi-Bug Triage）

**问题**：用户一次报告 4-5 个 bug，AI 按描述顺序从第一个开始修，但如果 bug 1 和 bug 3 共享根因，分开修会产生冲突补丁。

**方案**：
- 新增 P1.5 步骤，在 P1（问题理解）和 P2（复现）之间
- 列分诊表：症状 + 疑似类型 + 疑似共因 + 优先级
- 3 轴评估：用户影响 × 耦合度 × 信息依赖
- 共因 bug 聚为一组，修一次验证多个
- AskUserQuestion 确认执行顺序

### 改动 2：P3.1 — 标签化假设日志

**问题**：临时调试日志是 ad-hoc 的 `console.log('debug:', xxx)`，复现后不知道哪条日志对应哪个假设。

**方案**：
- 强制使用 `[DEBUG H{N}]` 命名规则
- 每条日志标注假设编号
- 复现后按标签过滤，快速判断哪个假设被证实/证伪
- 清理时搜索 `[DEBUG H` 批量删除

### 改动 3：P3.4 — 5 Whys 因果链格式

**问题**：根因解释是自由文本，质量不稳定。有时跳过因果关系直接给结论，用户只知道"什么出了问题"但不知道"为什么一步步走到这里"。

**方案**：
- P3.4 的根因解释改为结构化 5 Whys 链格式
- 每层用"为什么→因为"递进，链到根因即止
- 用用户可见的术语，不用代码术语
- 直接支持用户的"知识成长"目标

### 改动 4：Bug 类型分类枚举

**问题**：bugfix 文档没有类型字段，无法按类别聚合统计。回看历史时不知道"我们最常犯哪类错误"。

**方案**：
- P3.4 和 bugfix 文档模板新增 `类型` 字段
- 7 个分类（根据项目实际 bug 模式定制）：
  - `数据流断裂` — 数据在传递链路中断裂或变形
  - `前后端不一致` — 前端期望 vs 后端实际不匹配
  - `状态同步` — 多处状态未保持一致
  - `配置漂移` — 环境/配置差异导致的问题
  - `工具链约束` — 构建工具/框架的隐式规则被违反
  - `UI渲染` — 纯视觉/布局问题
  - `兄弟路径遗漏` — 新增逻辑时遗漏了同级别的其他路径
- 积累后可发现模式（如"70% 的 bug 是兄弟路径遗漏"）

### 改动 5：防护措施字段

**问题**：修完 bug 只写经验教训，但没有强制思考"怎么防止这类 bug 再出现"。

**方案**：
- bugfix 文档模板新增 `防护措施` 字段，三选一不能留空：
  - `回归测试` — 测试文件路径（P4.4 已覆盖）
  - `lint/约束规则` — 如新增 ESLint rule 或 CLAUDE.md R-rule
  - `架构改进` — 记录到 TODO，下次设计评审时处理
- 规则 16 强制执行
