# 复盘：文档驱动开发 Skill 生态构建

---

## 一、项目概览

- **我的需求**：构建一套 AI Skill 工作流，从需求诊断到设计、工程、QA 全流程自动调度，支持多会话并行开发不同功能
- **AI 做了什么**：创建 5 个 Skill（up-prd / do-dev / do-design / do-eng / do-qa），参考现有 cn-* 系列方法论补齐深度，引入 GSD 框架的上下文工程和 Wave 并行机制，设计 `.features/` 状态隔离架构 + Git Worktree 代码隔离方案
- **最终交付**：5 个 SKILL.md + 工具文档（README / 使用手册 / PRD / CHANGELOG / 多会话并行规范）+ symlink 注册

---

## 二、知识地图

### 领域 A：Skill 方法论设计

#### 模块 1：你遇到了什么问题

最初创建 do-* 系列 skill 时，只是把参考对象（cn-plan-eng / cn-plan-design / cn-qa / cn-deliver）的内容"压缩成口号"——比如工程思维从 7 条砍到 5 条，80 项设计审计清单完全丢了，范围挑战机制没有。结果 skill 看着有模有样，但实际执行时缺少可操作的判断基准，产出质量没保障。

#### 模块 2：关键概念速查表

| 概念 | 一句话解释 | 什么时候用 | 下次告诉 AI 的关键词 |
|------|-----------|-----------|-------------------|
| 方法论继承 | 参考 skill 的核心方法不是"参考"而是"继承"——精华不能压缩 | 基于现有 skill 创建新 skill 时 | "完整继承方法论，不要压缩" |
| 可操作判断基准 | "8+文件触发复杂度检查"比"注意复杂度"有用 100 倍 | 写任何审查流程时 | "每条规则要有具体的触发阈值" |
| 10 星挑战 | 问"10 星版本是什么"，然后评估差距是否值得追 | 所有审查层级，不只是深度审查 | "所有层级都做 10 星挑战" |
| 范围挑战 | 方案开始前先问"最小变更集是什么"，8+ 文件 = 警告 | 工程方案设计前 | "先做范围挑战再设计方案" |
| 四章审查 | 架构 → 代码质量 → 测试 → 性能，每章独立确认 | 工程审查阶段 | "四章工程审查，每章暂停确认" |

#### 模块 3：实践归因

##### 我的决策

- 要求"参考改造而非照搬"——结合自身需求调整 cn-* 方法论
  → 背后的诉求：cn-* 是通用的审查工具，我需要的是"文档驱动 + 半自动调度"的实现工具
- 要求"所有层级都有 10 星挑战"
  → 背后的诉求：轻量审查也不应该放过提升空间，AI 辅助下"完整"的成本接近于零
- 要求"do-dev 半自动"——框架性工作交给我确认
  → 背后的诉求：不信任全自动决策，关键方向需要人类把关

##### AI 的决策

- 首次创建 do-* 时过度压缩方法论，丢了关键内容
  → 考虑过的替代方案：完整照搬 / 压缩精简 / 选择性继承
  → 选择理由：追求"精简"，认为口号化足够
  → 潜在风险：口号没有可操作性，模型执行时会自由发挥——这正是出问题的地方
- Research 阶段最初设计为 4 个并行 Agent
  → 考虑过的替代方案：4 个并行 Agent / 2 个 Agent / 单 Agent / prompt 驱动
  → 选择理由：参考 GSD 框架的 research 模式，追求调研深度
  → 潜在风险：Token 消耗过高，耗时长——用户后来要求改为 prompt 驱动

##### 最终定稿

- 最终方案：5 个 skill 完整继承参考方法论的可操作内容，Research 改为 prompt 驱动（基于知识 + 代码扫描 + 按需 WebSearch），10 星挑战覆盖所有审查层级
- 与初始方案的差异：从"压缩精简"转向"完整继承 + 按需调整"，Research 从 4 Agent 降为当前上下文完成
- 当前稳定性：方法论层面稳定，但尚未在真实项目中跑完一轮完整流程验证

#### 模块 4：最佳实践

```
▍ 创建 Skill 时完整继承参考方法论
适用场景：基于已有 skill 创建变体 | 重要程度：⭐⭐⭐⭐⭐ | 本次用到：✅
```

**资源矩阵**

| 类型 | 资源 | 关键价值 |
|------|------|---------|
| 🏭 产品案例 | [GSD Framework](https://github.com/gsd-build/get-shit-done) | 展示了完整的 skill 生态如何设计——phase 拆分、context engineering、wave 执行 |
| 📐 权威规范 | cn-plan-eng / cn-plan-design / cn-qa SKILL.md | 定义了各领域审查的"完整版"应该包含什么 |

**知识合成**

- **核心洞察**：Skill 的价值不在于覆盖多少步骤，而在于每个步骤有没有可操作的判断基准。"注意复杂度"不是 skill，"8+ 文件触发警告"才是 skill。
- **权衡代价**：

  | 你得到了 | 你付出了 |
  |---------|---------|
  | 模型执行时有具体的判断锚点，产出质量稳定 | SKILL.md 文件更长，token 消耗更高 |
  | 每个审查步骤可追溯、可验证 | 首次创建时间更长 |

- **何时不用**：如果 skill 只是一个简单的格式转换器（如"把 JSON 转成表格"），不需要方法论深度

**复用接口**

- **决策清单**：
  - [ ] 参考 skill 的核心审查项是否全部继承？
  - [ ] 每条规则是否有具体的触发阈值或判断基准？
  - [ ] 是否有"不需要用户确认"的清单（避免过度交互）？
- **AI 词汇表**：`完整继承方法论` `可操作判断基准` `触发阈值` `不压缩口号化`
- **验收清单**：逐条对照参考 skill，确认关键条目未丢失

#### 模块 5：避坑指南

| 坑 | 触发条件 | 解决方案 |
|----|---------|---------|
| 方法论压缩成口号 | 要求 AI "精简"或"参考"已有 skill | 明确说"完整继承"，创建后逐条对比验证 |
| 交互流程图缺失 | Skill 流程复杂但没有可视化 | 在 SKILL.md 顶部加 ASCII 交互流程图，标注所有用户交互点 |
| Symlink 循环引用 | 批量创建 symlink 时路径错误 | 创建后用 `ls -la` 验证，检查是否指向自身 |

### 领域 B：上下文工程与 GSD 框架

#### 模块 1：你遇到了什么问题

随着会话变长，AI 的输出质量会下降（context rot）——做到第三个 skill 时明显感觉方案变粗糙了。另外，所有工作在同一个上下文窗口完成，调度器和执行器抢上下文空间。GSD 框架解决了这个问题但整体比较重，需要找到适合自己场景的平衡点。

#### 模块 2：关键概念速查表

| 概念 | 一句话解释 | 什么时候用 | 下次告诉 AI 的关键词 |
|------|-----------|-----------|-------------------|
| Context Rot | 上下文窗口填满后 AI 输出质量劣化 | 长会话中感觉 AI 变笨了 | "用独立上下文执行子任务" |
| 上下文工程 | 主调度器只做编排（~30%），子任务在新鲜上下文中执行 | do-dev 调度子 skill 时 | "Agent 独立上下文" |
| Wave 并行 | 无依赖的任务归为同一 wave 并行执行 | 工程实现阶段的任务拆分 | "分析依赖，按 wave 并行" |
| 原子 Git 提交 | 每个任务一个 commit，支持 bisect 和独立回滚 | 代码实现阶段 | "每个任务独立 git commit" |
| Prompt 驱动调研 | 在当前上下文中一次性完成调研，不拉子 agent | Research 阶段 | "prompt 驱动，不拉 agent" |

#### 模块 3：实践归因

##### 我的决策

- 要求 Research 从 4 个并行 Agent 改为 prompt 驱动
  → 背后的诉求：4 个 agent 太慢、太贵，大部分调研基于已有知识就够了
- 要求合并为单一工具（方案 B 而非 5 个独立工具）
  → 背后的诉求：降低认知负担，一套流程走到底

##### AI 的决策

- 引入 GSD 的 Discussion 阶段（偏好收集）
  → 考虑过的替代方案：直接跳到 Research / 合并到 Research 中 / 独立阶段
  → 选择理由：PRD 说了"做什么"，但"怎么做"有很多灰区需要在调研前明确
  → 潜在风险：如果灰区很少，Discussion 会变成多余的一步——所以加了"可跳过"机制
- do-dev 调度子 skill 时使用 Agent 工具而非直接执行
  → 考虑过的替代方案：同一会话顺序执行 / Agent 独立上下文 / Agent + worktree
  → 选择理由：GSD 的核心设计——避免 context rot
  → 潜在风险：Agent 之间传递信息只能通过文件，丢失了对话上下文

##### 最终定稿

- 最终方案：do-dev 主上下文只做编排，子 skill 通过 Agent 工具在独立上下文执行；Research 用 prompt 驱动；Discussion 可跳过
- 与初始方案的差异：Research 从 4 Agent → prompt 驱动，节省大量 token 和时间
- 当前稳定性：架构设计合理，但 Agent 间通过文件传递上下文的效果需要实际验证

#### 模块 4：最佳实践

```
▍ 调度器与执行器分离
适用场景：多步骤工作流中防止 context rot | 重要程度：⭐⭐⭐⭐⭐ | 本次用到：✅
```

**资源矩阵**

| 类型 | 资源 | 关键价值 |
|------|------|---------|
| 🏭 产品案例 | [GSD Framework README](https://github.com/gsd-build/get-shit-done/blob/main/README.zh-CN.md) | 完整展示了 context engineering 的实践——主 agent 30-40% 利用率，子 agent 全新窗口 |
| 📐 权威规范 | Claude Code Agent SDK `isolation: "worktree"` | 官方支持的代码级隔离方案 |

**知识合成**

- **核心洞察**：AI 工作流的瓶颈不是模型能力，而是上下文管理。主调度器越精简，子任务的输出质量越高。
- **权衡代价**：

  | 你得到了 | 你付出了 |
  |---------|---------|
  | 子任务在全新上下文中执行，质量稳定 | Agent 间只能通过文件传递信息，对话上下文丢失 |
  | 可并行执行无依赖任务 | 架构复杂度增加，调试更困难 |

- **何时不用**：简单任务（5 分钟内完成），上下文不会填满

**复用接口**

- **决策清单**：
  - [ ] 任务是否复杂到需要子 agent？（>3 个步骤 + >5 个文件 = 需要）
  - [ ] 子 agent 之间是否有依赖？（有 → 不同 wave，无 → 同一 wave）
  - [ ] 传递给子 agent 的上下文是否精简？（只传文件路径 + 变更摘要）
- **AI 词汇表**：`独立上下文执行` `Agent 工具` `Wave 并行` `文件路径传递上下文`
- **验收清单**：主调度器上下文利用率 <40%，子任务产出文档完整

#### 模块 5：避坑指南

| 坑 | 触发条件 | 解决方案 |
|----|---------|---------|
| Context rot 导致后期 skill 粗糙 | 同一会话中顺序执行多个 skill | 子 skill 用 Agent 独立上下文 |
| Agent 间信息丢失 | 只传了文件路径没传变更摘要 | prompt 中明确包含变更摘要 + 文件路径 |
| Research 阶段过重 | 4 个并行 Agent 调研 | prompt 驱动，按需 WebSearch |

### 领域 C：多会话并行架构

#### 模块 1：你遇到了什么问题

实际工作中会同时开多个 Claude Code 终端，每个处理不同功能方向。但所有代码在同一个 git 分支上，状态写在同一个领域文档里——两个会话同时改 PRD.md 或 DESIGN.md，后写的直接覆盖先写的。会话崩溃后状态永远卡在"进行中"。

#### 模块 2：关键概念速查表

| 概念 | 一句话解释 | 什么时候用 | 下次告诉 AI 的关键词 |
|------|-----------|-----------|-------------------|
| Feature 隔离 | 每个功能有独立的状态目录（`.features/{id}/`） | 多会话并行开发时 | ".features/ 按 feature 隔离状态" |
| Git Worktree | 同一个 repo 创建多个工作目录，各自独立分支 | 多会话修改同一项目代码 | "git worktree" "claude -w" |
| 孤儿检测 | heartbeat 超过 30 分钟未更新 = 疑似崩溃的会话 | do-dev 启动时 | "惰性孤儿检测，30 分钟心跳" |
| 版本号预留 | up-prd 创建 feature 时预留版本号，避免冲突 | 多会话同时 bump 版本 | "版本号预留机制" |
| _registry.md | 全局索引表，所有 feature 的 ID/版本/状态/心跳 | 跨会话感知谁在做什么 | "全局注册表" |

#### 模块 3：实践归因

##### 我的决策

- 提出多会话并行的需求场景，要求诊断架构是否适配
  → 背后的诉求：日常工作就是多终端同时做不同功能，需要系统性解决冲突

##### AI 的决策

- 设计 `.features/` + `_registry.md` + `status.md` 三层状态管理
  → 考虑过的替代方案：继续用领域文档内嵌状态 / 用数据库（SQLite）/ 用 .lock 文件
  → 选择理由：Markdown 文件人类可读、Git 可追踪、不需要额外依赖
  → 潜在风险：多会话同时写 `_registry.md` 仍有理论冲突，但概率极低（只在 skill 启动/完成时写）
- 推荐 Git Worktree 做代码隔离
  → 考虑过的替代方案：Git 分支（不隔离工作目录）/ 复制项目目录 / Git submodule
  → 选择理由：共享 .git 数据库，磁盘开销极小，Claude Code 原生支持 `claude -w`
  → 潜在风险：端口冲突、SQLite 锁、每个 worktree 需要独立安装依赖
- 惰性孤儿检测（不需要守护进程）
  → 考虑过的替代方案：后台 cron 心跳检测 / 文件锁 / 无检测
  → 选择理由：每个 skill 启动时检查一次即可，不需要额外进程
  → 潜在风险：如果没有新会话启动，孤儿会一直留着——但不影响功能，只是视觉噪音

##### 最终定稿

- 最终方案：`.features/` 状态隔离 + Git Worktree 代码隔离，双层互补
- 与初始方案的差异：从"领域文档内嵌状态"到"独立状态文件"，5 个 SKILL.md 全部重写了状态管理章节
- 当前稳定性：架构设计完成，文档齐全，但尚未在真实多会话场景中验证

#### 模块 4：最佳实践

```
▍ 状态与内容分离
适用场景：多 Agent/多会话并行操作同一项目 | 重要程度：⭐⭐⭐⭐⭐ | 本次用到：✅
```

**资源矩阵**

| 类型 | 资源 | 关键价值 |
|------|------|---------|
| 📐 权威规范 | [Git Worktree 官方文档](https://git-scm.com/docs/git-worktree) | 定义了 worktree 的创建、管理和清理 |
| 🏭 产品案例 | Claude Code `claude -w` / `Agent(isolation: "worktree")` | 展示了 AI 工具如何原生支持 worktree |
| 🏭 产品案例 | [GSD Framework](https://github.com/gsd-build/get-shit-done) | 展示了 branch-per-feature 在 AI 工作流中的实践 |

**知识合成**

- **核心洞察**："谁在做什么"（可见性）和"代码不冲突"（隔离性）是两个独立的问题，需要两层方案分别解决。`.features/` 解决前者，Git Worktree 解决后者。
- **权衡代价**：

  | 你得到了 | 你付出了 |
  |---------|---------|
  | 多会话完全隔离，互不干扰 | 架构复杂度增加，需要管理 worktree 生命周期 |
  | 状态可追踪，孤儿可检测 | 每次状态变更都要更新两个文件（status.md + _registry.md） |
  | 合并时 Git 自动处理大部分冲突 | 合并仍可能有冲突，需要人工介入 |

- **何时不用**：单会话开发、项目极小（<5 个文件）、不需要并行

**复用接口**

- **决策清单**：
  - [ ] 是否需要同时开发多个功能？（是 → 用 `.features/` + worktree）
  - [ ] 功能之间是否修改同一文件？（是 → 必须用 worktree）
  - [ ] 项目有数据库文件在 Git 中吗？（是 → 每个 worktree 用独立 db 实例）
- **AI 词汇表**：`.features/ 状态隔离` `git worktree` `claude -w` `_registry.md 注册表` `heartbeat 孤儿检测`
- **验收清单**：两个终端同时操作不同 feature，互不影响；崩溃后孤儿被检测到

#### 模块 5：避坑指南

| 坑 | 触发条件 | 解决方案 |
|----|---------|---------|
| 领域文档写冲突 | 两个会话同时写 PRD.md 顶部状态 | 状态移到 `.features/` 独立文件 |
| 状态孤儿 | 会话崩溃后状态卡在"进行中" | heartbeat + 惰性孤儿检测 |
| 版本号冲突 | 两个会话同时 bump 版本 | up-prd 创建 feature 时预留版本号 |
| 端口冲突 | 多个 worktree 同时跑 dev server | 每个 worktree 在 .env 中配不同端口 |
| SQLite 锁 | 多个 worktree 访问同一个 .db 文件 | 数据库文件不放 Git，或用独立实例 |
| Symlink 循环引用 | 在 skill 目录内创建指向自身的 symlink | 创建后 `ls -la` 验证目标路径 |

| 孤儿检测 | Orphan Detection | 心跳超时的 feature 可能是崩溃的会话 | do-dev 启动时 | "惰性孤儿检测，30 分钟心跳" |
| 心跳 | Heartbeat | 每次状态变更时更新时间戳 | 状态管理 | "每次写 status.md 都更新 heartbeat" |
| 范围挑战 | Scope Challenge | 方案前先问"最小变更集是什么" | 工程方案设计前 | "8+ 文件触发范围挑战" |
| 四章审查 | Four-Chapter Review | 架构→代码质量→测试→性能 | 工程审查阶段 | "四章工程审查，每章暂停确认" |
| 10 星挑战 | 10-Star Challenge | "如果这是 10 星产品会怎样" | 所有审查层级 | "当前几星？10 星版本是什么？" |
| 方法论继承 | Methodology Inheritance | 参考 skill 的关键审查项必须完整继承 | 创建新 skill 时 | "完整继承，不压缩口号化" |
| 灰区 | Gray Area | PRD 说了"做什么"但"怎么做"有多种选择 | Discussion 阶段 | "识别灰区，收集用户偏好" |
| 半自动调度 | Semi-Auto Orchestration | 列出建议供用户确认，而非全自动执行 | do-dev 调度子 skill | "半自动，关键决策交给我确认" |

## 四、下次协作指南

### Prompt 模板

**模板 1：创建新 Skill**
> 基于 {参考 skill 名} 创建一个新的 skill "{新 skill 名}"。要求：
> 1. 完整继承参考 skill 的方法论，不压缩口号化
> 2. 每条审查规则必须有具体的触发阈值
> 3. 在 SKILL.md 顶部加交互流程图，标注所有用户交互点
> 4. 使用 .features/ 架构管理运行状态
> 创建后，逐条对比参考 skill 确认关键内容未丢失。

**模板 2：诊断现有架构**
> 我的使用场景是 {描述场景}。帮我诊断当前这套 {系统名} 是否适配这种场景。要求：
> 1. 列出硬伤（致命问题）和软伤（可接受问题）
> 2. 给出解决方案，用交互图展示完整架构
> 3. 展示数据结构和读写时序

**模板 3：多会话并行开发**
> 我要在 {项目名} 上同时开发 {功能 A} 和 {功能 B}。请按以下流程执行：
> 1. 在终端 1 执行 `claude -w feat-{功能A}`
> 2. 在终端 2 执行 `claude -w feat-{功能B}`
> 3. 每个终端内用 `/up-prd` 开始，`.features/` 会自动创建和管理

### 关键决策点（不能全交给 AI）

1. **功能方向和优先级** — AI 不知道你的业务上下文，功能做不做、先做哪个必须你定
2. **设计风格偏好** — Discussion 阶段的灰区问题必须你回答
3. **范围取舍** — 范围挑战触发时（8+ 文件），缩减还是继续必须你决定
4. **合并策略** — 多个 worktree 完成后，哪个先合、冲突怎么解必须你把关
5. **上线决策** — QA 报告出来后，有遗留 Bug 是否上线必须你判断
