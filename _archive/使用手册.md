# 文档驱动开发工作流 — 完整使用手册

> 一套 5 个 AI Skill 组成的文档驱动开发体系：从需求诊断到设计、工程、QA 全流程自动调度

---

## 工具清单

| Skill | 命令 | 定位 | 产出文档 | 参考方法论 |
|-------|------|------|---------|-----------|
| **up-prd** | `/up-prd` | 产品诊断与 PRD 迭代 | PRD.md + PRD-CHANGELOG | cn-plan-product |
| **do-dev** | `/do-dev` | 开发调度器（编排层） | CONTEXT.md + RESEARCH.md | cn-deliver + GSD |
| **do-design** | `/do-design` | 设计文档 + 样式实现 | DESIGN.md + DESIGN-CHANGELOG | cn-plan-design |
| **do-eng** | `/do-eng` | 工程文档 + 代码实现 | ENGINEERING.md + ENG-CHANGELOG | cn-plan-eng |
| **do-qa** | `/do-qa` | QA 文档 + 测试执行 | QA.md + QA-CHANGELOG | cn-qa |

## 整体架构

```
用户描述问题/需求
       │
       ▼
  ┌─────────┐
  │  up-prd  │ ← 产品诊断：三层自适应审查 + CHANGELOG 热点分析
  │          │   问题归因（设计缺陷/实现偏离/PRD遗漏）
  └────┬─────┘   反驳机制 + 10 星挑战
       │
       │ PRD 迭代摘要 + 迭代交付说明
       ▼
  ┌─────────┐
  │  do-dev  │ ← 编排层：Discussion → Research → 调度
  │          │   上下文工程：子 skill 在独立上下文执行
  └────┬─────┘   半自动：列计划 → 用户确认 → 执行
       │
       │ Agent 工具（独立上下文，避免 context rot）
       │
  ┌────┼────────────────┐
  ▼    ▼                ▼
┌──────┐ ┌──────┐ ┌──────┐
│design│ │ eng  │ │  qa  │
│      │ │      │ │      │
│12认知│ │7思维 │ │5维度 │
│80审计│ │4章审 │ │3级测 │
│A-F分 │ │Wave  │ │原子  │
│      │ │原子  │ │修复  │
│      │ │commit│ │      │
└──────┘ └──────┘ └──────┘
```

## 核心设计理念

### 1. 文档驱动

先写文档明确需求，再基于文档实现。PRD 是唯一需求源头，所有下游 skill 从 PRD 获取输入。

### 2. 上下文工程（来自 GSD 框架）

解决 **context rot** 问题——随着 AI 上下文窗口被填满，输出质量逐步劣化。

```
传统方式：所有工作在同一上下文 → 质量随时间下降
本系统：主调度器只做编排（~30-40% 上下文利用率）
       → 子 skill 通过 Agent 工具获得全新上下文
       → 真正的工作发生在子 agent 的新鲜上下文中
```

### 3. 半自动调度

关键决策交用户确认，具体执行自动完成。do-dev 有 3 个硬卡点：Discussion 后 / 执行计划确认 / 交付总结。

### 4. 原子化交付

每个任务独立 git commit，支持 `git bisect` 定位问题、独立回滚、清晰的 git 历史。

## 使用场景与操作指南

### 场景 1：发现问题，完整修复

最完整的流程。适合功能异常、体验问题、需求调整等。

```bash
# Step 1：诊断问题，更新 PRD
# 在对话中描述问题，或直接调用
/up-prd
# 告诉它："info-feed 的分类准确率太低，37% 的内容被归为其他"
# AI 会：诊断根因 → 设计方案 → 与你确认 → 更新 PRD v3.2

# Step 2：进入开发
/do-dev
# AI 会：
#   1. Discussion — 问你布局偏好、API 格式等灰区问题
#   2. Research — 调研技术方案（基于知识+代码扫描+按需搜索）
#   3. 列出执行计划 — "建议 do-eng → do-qa，纯后端改动无需设计"
#   4. 你确认后 — 在独立上下文中调度子 skill 执行
```

### 场景 2：PRD 已更新，直接开发

跳过诊断环节，直接进入开发调度。

```bash
/do-dev
# AI 读取最新 PRD 迭代摘要 → Discussion → Research → 调度
```

### 场景 3：只做某一环

不需要走完整流程，单独执行某个环节。

```bash
/do-design    # 只更新设计文档 + 实现样式
/do-eng       # 只更新工程文档 + 实现代码
/do-qa        # 只更新测试文档 + 执行测试
```

### 场景 4：新项目从零建立文档体系

项目已有代码但没有任何文档。

```bash
# 按顺序创建（每个 skill 会深度阅读代码 + 多轮交互确认）
/up-prd       # → 深度读代码 → 5 轮确认 → 创建 PRD v1.0
/do-design    # → 从 CSS 提取设计体系 → 创建 DESIGN.md v1.0
/do-eng       # → 分析架构 → 5 轮确认 → 创建 ENGINEERING.md v1.0
/do-qa        # → 建立测试基线 → 创建 QA.md v1.0
```

## 各 Skill 详细说明

### `/up-prd` — 产品诊断与 PRD 迭代

**何时用**：遇到产品问题、需要调整需求、新增功能、砍掉功能

**核心流程**：

```
第0步：定位 PRD + CHANGELOG（模式匹配，不写死文件名）
第1步：理解现状（读 PRD + CHANGELOG + 源码分析）
第2步：诊断与审查（三层自适应深度）
第3步：方案确认（多轮交互）
第4步：写入文档（CHANGELOG + PRD 更新 + 迭代交付说明）
```

**三层自适应审查**：

| 层级 | 触发条件 | 做什么 |
|------|---------|-------|
| 轻量 | 单个小改动（改阈值、调文案） | 定位问题 → 确认方案 → 更新 |
| 标准 | 功能调整或多个小改动 | 归因分析 + 模块健康度检查 + 方案对比 |
| 深度 | 新模块、架构调整、或 CHANGELOG 显示反复修改 | 假设审查 + 10 星挑战 + 反驳机制 |

**自动升级规则**：
- 多个小需求集中在同一模块 → 轻量升级到标准
- CHANGELOG 中同一模块 ≥3 个版本反复修改 → 自动升级到深度

**诊断输出**：
- 问题归因：产品设计缺陷 / 实现偏离 PRD / PRD 遗漏场景
- 方案建议：新增功能 / 优化现有功能 / 砍掉不需要的功能 / 调整架构
- 反驳意见：如果需求放在这里不合适，直接说出来并给替代方案

**PRD 文件结构**：

```markdown
# 项目名 — PRD

**版本:** vX.Y | **日期:** YYYY-MM-DD

## 迭代历史摘要（保留所有版本）
### vX.Y — [变更概览] — 详见 CHANGELOG
### vX.Y-1 — [变更概览]
...

## 一、产品定位
## 二、信息架构
## 三、核心功能详细设计    ← 正文用 [vX.Y 新增/修改] 标记
...
```

### `/do-dev` — 开发调度器

**何时用**：PRD 更新后，需要协调设计、工程、QA 执行

**核心流程**：

```
第0步：读取 PRD 输出 + 判断项目类型（frontend/backend/fullstack）
第1步：Discussion（结构化偏好收集）
第2步：Research（四维技术调研）
第3步：分析与调度建议（用户确认）
第4步：Wave 并行调度（独立上下文执行）
第5步：汇总交付
```

**Discussion 阶段**：

在调研前收集用户脑中的偏好，避免 AI 猜测。

| 变更类型 | 讨论的灰区 |
|---------|-----------|
| 视觉功能 | 布局偏好、信息密度、交互方式、空状态处理 |
| API / CLI | 返回格式、错误码设计、参数命名 |
| 数据处理 | 分组标准、去重策略、命名规则 |

产出 `CONTEXT.md`，可跳过（说"用默认"）。

**Research 阶段**：

在当前上下文中一次性完成四个维度的调研（不拉子 agent，通过 prompt 驱动）：

| 维度 | 调研内容 | 手段 |
|------|---------|------|
| 技术栈 | 最佳实践、库选择、兼容性 | 基于知识 + 按需 WebSearch |
| 架构模式 | 数据流设计、模块划分 | 基于知识 + 项目代码分析 |
| 坑点与风险 | 已知陷阱、规避策略、边界情况 | 基于知识 + 按需 WebSearch |
| 可复用资产 | 项目中已有的模块/函数/模式 | Glob/Grep/Read 扫描项目代码 |

产出 `RESEARCH.md`，可跳过（纯配置变更）。

**调度规则**：

| 变更类型 | 调度的子 skill |
|---------|--------------|
| 纯视觉调整 | do-design |
| 新增/修改交互 | do-design → do-eng |
| 纯后端逻辑 | do-eng |
| 涉及 UI 的功能变更 | do-design → do-eng → do-qa |
| 新功能模块（全栈） | do-design → do-eng → do-qa |
| Bug 修复 | do-eng → do-qa |

**三种模式**：

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| 交互（默认） | 3 个硬卡点暂停确认 | 重要功能、需要把控方向 |
| 自动（`--auto`） | 前置沟通后全自动，不 commit | 信任 AI 判断的小改动 |
| 恢复（`--resume`） | 从 `.do-dev/state.json` 恢复 | 上次中断后继续 |

### `/do-design` — 设计文档与实现

**何时用**：需要更新设计规范、调整样式、新增页面布局

**方法论**（来自 cn-plan-design）：
- **12 条认知模式**：同理心即模拟、约束崇拜、提问反射、边界情况偏执 等
- **80 项审计清单**（10 类）：视觉层级、排版、颜色对比度、间距布局、交互状态、响应式、动效、微文案、AI 模板痕迹（10 个反面模式）、性能即设计
- **A-F 评分体系**：10 类权重评估
- **设计批评格式**："我注意到..."结构化反馈

**核心流程**：

| 模式 | 流程 |
|------|------|
| 0→1 | 从 CSS 提取设计体系 → 读 PRD 设计章节 → 多轮确认 → DESIGN.md |
| 迭代 | 读 PRD 变更 → 分析设计影响 → ASCII 线框图确认 → 更新 + 实现 |

**7 条核心规则**：
1. 避免 AI 模板痕迹（紫色渐变、三列网格等）
2. 响应式不是可选的
3. 遵循现有设计体系
4. 不要过度设计——最少的改动实现目标
5. 颜色对比度必须满足 WCAG AA
6. 动效服务于功能，不是装饰
7. 空状态和错误状态必须设计

### `/do-eng` — 工程文档与代码实现

**何时用**：需要写代码、修改架构、实现功能

**方法论**（来自 cn-plan-eng）：

**7 条工程思维框架**：
1. **爆炸半径直觉** — 最坏情况是什么？影响多少系统和人？
2. **无聊优先** — 每个公司大约只有三个创新机会，其他用成熟技术
3. **增量而非革命** — 绞杀者模式，灰度发布，重构而非重写
4. **系统而非英雄** — 为凌晨3点疲惫的工程师设计
5. **可逆性偏好** — 功能开关、A/B测试，让犯错代价变低
6. **DX就是产品质量** — 慢 CI = 更差的软件
7. **先让修改变简单** — 先重构再实现，永不同时改结构和行为

**核心流程**：

```
第0步：定位文档（PRD + DESIGN.md + RESEARCH.md + ENGINEERING.md）
第1步：范围挑战
       → 8+ 文件或 2+ 新类 = 警告信号，建议缩减
       → TODOS 交叉检查
第2步：理解现状 / 从零创建文档
第3步：四章工程审查
       → 架构 → 代码质量 → 测试 → 性能
       → 每章发现问题单独 AskUserQuestion
第4步：工程方案设计（ASCII 架构图 + 技术选型 + 实现清单）
第5步：更新文档（ENGINEERING.md + CHANGELOG）
第6步：任务拆分与 Wave 规划
第7步：Wave 并行执行 + 原子 Git 提交
第8步：必需产出（3 个清单）
第9步：确认与总结
```

**任务拆分 + Wave 并行**：

```
                    Wave 1（并行）         Wave 2（依赖W1）    Wave 3
                   ┌──────────┐ ┌────────┐ ┌──────────┐      ┌──────┐
                   │ Task-01  │ │Task-02 │ │ Task-03  │      │T-05  │
                   │ 数据模型 │ │配置文件│ │ API 端点 │      │前端  │
                   └──────────┘ └────────┘ └────┬─────┘      │集成  │
                        │           │           │ ┌────────┐ └──────┘
                        └───────────┴───────────┘ │Task-04 │    ↑
                               依赖               │前端组件│────┘
                                                  └────────┘
```

规则：
- **垂直切片优先** — 端到端完成功能，而非水平分层（先所有 model 再所有 API）
- **每个任务 1-3 个文件** — 足够小，可在单个上下文窗口完成
- **每个任务可独立验证** — 有明确的验证命令
- **原子 Git 提交** — 每个任务完成后立刻 `feat(scope): task name`

**3 个必需产出**：
1. **未纳入范围清单** — 考虑但明确推迟的工作
2. **已存在内容清单** — 复用了还是重复建设
3. **故障模式清单** — 每个新代码路径的失败方式（无测试+无错误处理+静默失败=标红）

### `/do-qa` — QA 文档与测试执行

**何时用**：代码实现后需要测试验证

**方法论**（来自 cn-qa）：

**5 维度健康评分**（起始 100 分，按 Bug 扣分）：

| 维度 | 权重 | 检查内容 |
|------|------|---------|
| 功能完整性 | 30% | 核心功能是否正常 |
| UI/UX 一致性 | 25% | 视觉和交互是否符合设计 |
| 性能表现 | 20% | 加载速度、响应时间 |
| 错误处理 | 15% | 边界情况、异常场景 |
| 兼容性 | 10% | 不同浏览器/设备 |

**核心流程**：

| 模式 | 流程 |
|------|------|
| 0→1 | 从 PRD 提取验收标准 → 从 ENGINEERING.md 提取测试矩阵 → 建立基线 |
| 迭代 | 读取变更范围 → 制定测试计划 → 确认策略 → 执行测试 |

**三级测试**：

| 级别 | 覆盖范围 | 适用场景 |
|------|---------|---------|
| 快速 | 仅严重问题（功能不可用、数据丢失） | 小改动、紧急修复 |
| 标准 | + 中等问题（功能异常、交互问题） | 日常迭代 |
| 详尽 | + 外观和边界情况 | 重大版本、上线前 |

**测试模式**：
- **浏览器测试** — 无头浏览器 E2E（页面加载/核心功能/表单输入/错误状态/响应式）
- **纯代码测试** — 运行项目测试框架

**Bug 处理**：
- Bug 登记格式：现象 / 步骤 / 截图 / 根因
- 原子提交修复：每个 Bug 独立 commit + 验证
- 修复后全量回归测试

**最终产出**：
- Bug 记录（严重/高/中/低分级）
- 健康评分（0-100）
- 上线就绪判定（✅ 可上线 / ⚠️ 需关注 / ❌ 不建议）

## 项目文档体系

每个使用本工作流的项目，docs/ 目录最终结构如下：

```
project/
├── docs/
│   ├── PRD.md                        ← up-prd 管理
│   ├── PRD-CHANGELOG.md              ← up-prd 管理
│   ├── vX.Y-CONTEXT.md               ← do-dev 管理（用户偏好）
│   ├── vX.Y-RESEARCH.md              ← do-dev 管理（技术调研）
│   ├── DESIGN.md                     ← do-design 管理
│   ├── DESIGN-CHANGELOG.md           ← do-design 管理
│   ├── ENGINEERING.md                ← do-eng 管理（前后端合一）
│   ├── ENGINEERING-CHANGELOG.md      ← do-eng 管理
│   ├── QA.md                         ← do-qa 管理
│   └── QA-CHANGELOG.md               ← do-qa 管理
│
├── .features/                          ← Feature 状态管理（按 feature 隔离）
│   ├── _registry.md                    # 全局索引（所有 feature 一览）
│   ├── dedup-optimization/             # 功能 A
│   │   ├── status.md                   # Pipeline + Tasks + QA Items 状态
│   │   └── state.json                  # do-dev 调度状态
│   └── channel-mgmt-v2/               # 功能 B
│       ├── status.md
│       └── state.json
│
├── .do-dev/                            ← do-dev 调度状态（兼容旧流程）
│   ├── checkpoints/                    # 阶段检查点（patch 文件）
│   │   ├── design-done.patch
│   │   ├── eng-done.patch
│   │   └── qa-done.patch
│   └── delivery-report.md             # 交付报告
│
└── ...（项目源码）
```

**关键规则**：
- **领域文档只存内容，不存运行状态** — 运行状态写入 `.features/{id}/status.md`
- 文件名不固定，所有 skill 通过模式匹配定位（支持中文命名）
- 每份文档顶部有迭代历史摘要，保留所有版本记录
- CHANGELOG 不存在时从 git 历史 + 项目文档回溯创建
- 下游 skill 读取上游文档（do-eng 读 DESIGN.md，do-qa 读 ENGINEERING.md）
- 下游 skill 通过 `.features/{id}/status.md` 感知上游完成状态

### 多会话并行支持

`.features/` 架构的核心设计就是支持多会话并行：

- **按 feature 隔离**：每个 feature 有独立的 status.md，不同会话操作不同 feature 互不干扰
- **孤儿检测**：heartbeat 超过 30 分钟未更新 → 疑似孤儿，由 do-dev 启动时惰性检测
- **版本号预留**：up-prd 创建 feature 时预留版本号，避免多会话冲突
- **配合 Git Worktree**：推荐每个 feature 在独立 worktree 中开发，彻底消除代码冲突（详见[多会话并行规范](多会话并行规范.md)）

| do-dev | cn-deliver + GSD | 状态管理、双模式、检查点、上下文工程、Wave 并行 |

上下文工程和 Wave 并行执行机制参考了 [GSD (Get Shit Done)](https://github.com/gsd-build/get-shit-done) 框架的核心设计。

## 常见问题

**Q：必须按顺序执行吗？**
A：不是。每个 skill 独立可用。调度器只是建议最佳顺序。

**Q：没有 PRD 能直接用 do-eng 吗？**
A：可以。do-eng 会从代码推断上下文，但有 PRD 效果更好。

**Q：do-dev 会自动执行所有子技能吗？**
A：不会。do-dev 是半自动的，列出建议后等你确认才执行。

**Q：CHANGELOG 文件名必须是英文吗？**
A：不必须。所有 skill 通过模式匹配，支持 `变更日志.md`、`CHANGELOG.md` 等。

**Q：什么是 context rot？**
A：随着 AI 上下文窗口被填满，输出质量逐步劣化。do-dev 让子 skill 在独立上下文中执行来解决。

**Q：原子 Git 提交有什么好处？**
A：`git bisect` 精确定位 bug、独立 revert、清晰的 git 历史。

**Q：Discussion 和 Research 可以跳过吗？**
A：可以。说"用默认"跳 Discussion，"跳过调研"跳 Research。

**Q：do-dev 中断了怎么恢复？**
A：加 `--resume`，从 `.do-dev/state.json` 恢复到中断阶段。

**Q：多个会话同时开发不同功能会冲突吗？**
A：不会。每个功能通过 `.features/{id}/status.md` 独立跟踪状态，领域文档不含运行状态。配合 Git Worktree 使用可彻底消除代码冲突。

**Q：会话崩溃后状态会卡在"进行中"吗？**
A：do-dev 启动时会检查 `_registry.md`，heartbeat 超过 30 分钟的 feature 会触发孤儿检测警告，你可以选择认领、标记废弃或忽略。
