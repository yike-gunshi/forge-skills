# 文档驱动开发工作流 — 使用说明

## 概述

这套 skill 体系实现了**文档驱动的开发流程**：用户描述问题或需求 → 自动诊断并更新产品文档 → 调度设计、工程、QA 逐层落地。每个 skill 管理自己的领域文档 + 变更日志，既可被调度器串联，也可独立调用。

### 整体架构

```
用户描述问题/需求
       │
       ▼
  ┌─────────┐
  │  up-prd  │  产品诊断 → 更新 PRD + CHANGELOG
  └────┬─────┘
       │ PRD 迭代摘要
       ▼
  ┌─────────┐
  │  do-dev  │  Discussion → Research(四维调研) → 调度建议 → 用户确认
  └────┬─────┘
       │ 独立上下文执行（解决 context rot）
  ┌────┼────────────────┐
  ▼    ▼                ▼
┌──────┐ ┌──────┐ ┌──────┐
│design│ │ eng  │ │  qa  │
└──────┘ └──────┘ └──────┘
  设计     工程      QA
  文档+    文档+    文档+
  样式     代码     测试
  实现     实现     执行
```

### 核心设计理念

1. **文档驱动** — 先写文档明确需求，再基于文档实现。PRD 是唯一需求源头
2. **上下文工程** — 子 skill 在独立上下文中执行（Agent 工具），主调度器只做编排，避免 context rot（质量随上下文填满而劣化）
3. **半自动调度** — 关键决策交用户确认，具体执行自动完成
4. **原子化交付** — 每个任务独立 git commit，可追溯、可回滚、可 bisect

## 使用场景

### 场景 1：完整流程（推荐）

用户发现问题或有新需求，走完整链路：

```
1. 用户：「瀑布流布局在移动端显示有问题，卡片间距不对」
2. /up-prd             → 诊断根因，更新 PRD v3.2
3. /do-dev             → Discussion（收集偏好）→ Research（技术调研）→ 调度建议
4. 用户确认执行计划
5. do-dev 在独立上下文中依次调度 do-design → do-eng → do-qa
```

### 场景 2：跳过诊断，直接开发

PRD 已经更新好，直接进入开发：

```
1. /do-dev             → 读取最新 PRD 迭代摘要，建议调度方案
2. 用户确认
3. 执行
```

### 场景 3：单独调用子技能

只需要做设计、只需要写代码、或只需要跑测试：

```
/do-design             → 单独管理设计文档 + 实现样式
/do-eng                → 单独管理工程文档 + 实现代码
/do-qa                 → 单独管理测试文档 + 执行测试
```

### 场景 4：从零创建项目文档

项目还没有任何文档，需要从零建立：

```
1. /up-prd             → 深度阅读代码，多轮交互，创建 PRD v1.0
2. /do-design          → 从代码提取设计体系，创建 DESIGN.md v1.0
3. /do-eng             → 分析架构，创建 ENGINEERING.md v1.0
4. /do-qa              → 建立测试基线，创建 QA.md v1.0
```

## 各 Skill 详细说明

### /up-prd — 产品诊断与 PRD 迭代

**触发词**：「更新PRD」「调整需求」「迭代PRD」或描述产品问题

**核心机制**：

- **三层自适应审查**：根据变更大小自动选择深度
  - 轻量（改阈值、调文案）→ 标准（功能调整、多个小改动）→ 深度（新模块、架构调整）
  - 自动升级：CHANGELOG 中同一模块 ≥3 个版本反复修改 → 触发深度审查
- **问题归因**：区分产品设计缺陷 / 实现偏离 PRD / PRD 遗漏场景
- **反驳机制**：认为需求不合理时直接说出来并给替代方案（借鉴 cn-plan-product 的 10 星挑战）
- **双模式**：
  - 迭代：已有 PRD → 读取 PRD + CHANGELOG + 源码 → 诊断 → 方案确认 → 更新文档
  - 从零创建：没有 PRD → 深度读代码 + 5 轮交互确认 → 产出完整 PRD

**用户需要确认的**：诊断结论、PRD 修改方案
**不需要确认的**：文档格式、章节安排

**产出**：
- PRD.md（头部含所有版本迭代摘要 + 正文融入变更）
- PRD-CHANGELOG.md（完整决策历史：时间、背景、需求、方案、关键讨论）

### /do-dev — 开发调度器

**触发词**：「开始开发」「实现需求」

**核心机制**：

**第0步：读取 PRD 输出**
- 定位 PRD + CHANGELOG
- 判断项目类型：frontend / backend / fullstack

**第1步：Discussion 阶段（结构化偏好收集）**
- 在调研和规划前，识别"灰区"并收集用户偏好
- 灰区类型：视觉功能（布局、信息密度）/ API 设计（返回格式、错误码）/ 数据处理（分组、去重）等
- 产出 `CONTEXT.md`，传递给后续 Research 和子 skill
- 可跳过（用户说"用默认"）

**第2步：Research 阶段（四维调研，prompt 驱动）**
- 当前上下文一次性完成四维调研（不拉子 agent）：
  - 技术栈最佳实践（基于知识 + 按需 WebSearch）
  - 架构模式推荐（基于知识 + 代码分析）
  - 坑点和风险预警（基于知识 + 按需 WebSearch）
  - 可复用代码分析（Glob/Grep/Read 扫描项目）
- 产出 `RESEARCH.md`，供 do-design / do-eng 参考
- 可跳过（纯配置变更或用户明确跳过）

**第3步：分析与调度建议**
- 结合 PRD + RESEARCH.md，判断需要哪些子 skill

| 变更类型 | 需要的子技能 |
|---------|------------|
| 纯视觉调整（配色/间距/字体） | do-design |
| 新增/修改交互 | do-design → do-eng |
| 纯后端逻辑 | do-eng |
| 涉及 UI 的功能变更 | do-design → do-eng → do-qa |
| 新功能模块 | do-design → do-eng → do-qa |
| Bug 修复 | do-eng → do-qa |

- 列出执行计划（含调研摘要 + 风险提示），等待用户确认

**第4步：Wave 并行调度（上下文工程）**
- 使用 Agent 工具在**独立上下文**中执行每个子 skill
- 主调度器只传递文档路径 + 变更摘要，不传历史会话
- 子 skill 拥有全新上下文窗口（~200K token），避免 context rot
- 按依赖关系分 wave：do-design → do-eng → do-qa

**第5步：汇总交付**
- 输出交付报告：各阶段状态 + 代码变更统计 + 测试结果 + 健康评分 + 上线就绪判定
- 输出验收操作清单（具体操作步骤 + 预期结果 + 回归检查）

**模式**：
- 交互模式（默认）：3 个硬卡点暂停（Discussion 后 / 执行计划确认 / 交付总结）
- 自动模式（`--auto`）：前置沟通后全自动，检查点用 patch 保存
- 恢复模式（`--resume`）：从 `.do-dev/state.json` 恢复中断的流水线

### /do-design — 设计文档与实现

**触发词**：「设计」「更新设计」

**核心机制**：
- 参考 `/cn-plan-design` 方法论：12 条认知模式 + 80 项审计清单（10 类） + A-F 评分体系
- 审计覆盖：视觉层级、排版、颜色对比度、间距布局、交互状态、响应式、动效、微文案、AI 模板痕迹、性能即设计
- 设计批评格式："我注意到..."结构化反馈
- 7 条核心规则（如：避免 AI 模板痕迹、响应式不是可选的）

**0→1 模式**（项目无 DESIGN.md）：
1. 从 CSS/样式文件提取现有设计体系
2. 从 PRD 设计章节读取规范
3. 多轮确认（设计体系 → 页面布局 → 组件规范 → 交互方案）
4. 生成 DESIGN.md + DESIGN-CHANGELOG

**迭代模式**（已有 DESIGN.md）：
1. 读取 PRD + RESEARCH.md 最新变更
2. 分析对设计的影响
3. 输出 ASCII 线框图供用户确认
4. 确认后更新 DESIGN.md + 实现样式变更

**用户需要确认的**：线框图、配色方案、布局方向
**不需要确认的**：具体 CSS 实现细节

### /do-eng — 工程文档与代码实现

**触发词**：「工程」「实现」「写代码」

**核心机制**：
- 参考 `/cn-plan-eng` 方法论：7 条工程思维框架 + 完整性原则（"把湖烧干"）
- 范围挑战（第1步）：复杂度检查（8+ 文件或 2+ 新类 = 警告信号）
- 四章审查（第3步）：架构 → 代码质量 → 测试 → 性能，每章发现问题单独确认
- 三个必需产出：未纳入范围清单 + 已存在内容清单 + 故障模式清单

**0→1 模式**（项目无 ENGINEERING.md）：
1. 分析项目目录结构和技术栈
2. 5 轮确认（架构概览 → 技术栈 → 模块划分 → 数据流 → 技术债）
3. 生成 ENGINEERING.md（前后端合一）+ ENGINEERING-CHANGELOG

**迭代模式**（已有 ENGINEERING.md）：
1. 读取 PRD + DESIGN.md + RESEARCH.md 最新变更
2. 范围挑战（复杂度检查、复用检查、TODOS 交叉检查）
3. 四章工程审查（架构 → 代码质量 → 测试 → 性能）
4. 工程方案设计（ASCII 架构图 + 技术选型 + 实现清单 + API 设计 + 测试矩阵）
5. 用户确认后更新 ENGINEERING.md + ENGINEERING-CHANGELOG
6. **任务拆分**：将实现拆为原子任务
   - 垂直切片优先（端到端完成功能，而非水平分层）
   - 每个任务涉及 1-3 个文件，有明确验证方式
   - 结构化定义：文件列表 + 实现要求 + 验证方式 + 完成标准
7. **Wave 并行执行**：
   - 分析任务依赖关系，独立任务归入同一 wave 并行执行
   - 同一 wave 内使用 Agent 工具并行启动，每个任务在独立上下文中完成
   - Wave 间验证：运行验证命令 + 检查冲突 + 自动修复失败任务
8. **原子 Git 提交**：
   - 每个任务完成后立刻独立 commit
   - 格式：`feat(scope): task name`（支持 feat/fix/refactor/docs/test/chore）
   - 支持 `git bisect`、独立回滚、清晰的自动化可观测性

**用户需要确认的**：架构决策、技术选型、实现方案、任务拆分和 Wave 规划
**不需要确认的**：具体代码写法、变量命名

### /do-qa — QA 文档与测试执行

**触发词**：「测试」「QA」

**核心机制**：
- 参考 `/cn-qa` 方法论：5 维度健康评分（权重明确） + 4 条核心规则
- 浏览器测试具体命令（页面加载/核心功能/表单输入/错误状态/响应式）
- Bug 登记格式（现象/步骤/截图/根因）
- 原子提交修复流程（每个 Bug 独立 commit + 验证）
- 回归测试环节（修复后全量回归）

**0→1 模式**（项目无 QA.md）：
1. 从 PRD 提取验收标准
2. 从 ENGINEERING.md 提取测试矩阵
3. 建立测试基线和回归测试清单
4. 生成 QA.md + QA-CHANGELOG

**迭代模式**（已有 QA.md）：
1. 读取最新变更影响范围
2. 制定测试计划（3 级）：
   - **快速**：仅严重问题
   - **标准**：+ 中等问题
   - **详尽**：+ 外观和边界情况
3. 输出测试策略供用户确认
4. 确认后执行测试

**测试模式**：
- **浏览器测试**：使用无头浏览器进行 E2E 测试（适合前端变更）
- **纯代码测试**：运行项目测试框架（适合后端逻辑）

**用户需要确认的**：测试级别、测试策略、是否使用浏览器
**不需要确认的**：具体测试用例执行

**产出**：
- Bug 记录（严重/高/中/低分级）
- 健康评分（0-100，起始100分按 Bug 扣分）
- 上线就绪判定（✅ 可上线 / ⚠️ 需关注 / ❌ 不建议）

## 文档体系

### 项目文档结构

每个 skill 维护的文档遵循统一规范：

```
project/docs/
├── PRD.md                        ← up-prd 管理
├── PRD-CHANGELOG.md              ← up-prd 管理
├── vX.Y-CONTEXT.md               ← do-dev 管理（Discussion 产出）
├── vX.Y-RESEARCH.md              ← do-dev 管理（Research 产出）
├── DESIGN.md                     ← do-design 管理
├── DESIGN-CHANGELOG.md           ← do-design 管理
├── ENGINEERING.md                ← do-eng 管理
├── ENGINEERING-CHANGELOG.md      ← do-eng 管理
├── QA.md                         ← do-qa 管理
└── QA-CHANGELOG.md               ← do-qa 管理
```

### 状态管理（do-dev 调度器）

```
project/.do-dev/
├── state.json                    # 调度状态（持久化）
├── checkpoints/
│   ├── design-done.patch         # 设计阶段完成后的代码状态
│   ├── eng-done.patch            # 工程阶段完成后的完整 diff
│   └── qa-done.patch             # QA 修复后的完整 diff
└── delivery-report.md            # 最终交付报告
```

### 关键规则

- **文件名不固定** — 所有 skill 通过模式匹配定位文档（支持中文命名等变体）
- **迭代历史摘要** — 每份文档顶部保留所有版本记录
- **CHANGELOG 自动创建** — 不存在时从 git 历史 + 项目文档回溯创建
- **上下游传递** — 下游 skill 读取上游文档（do-eng 读 DESIGN.md，do-qa 读 ENGINEERING.md）
- **上下文隔离** — do-dev 调度时，子 skill 在独立 Agent 上下文中执行

## 常见问题

**Q：必须按顺序执行吗？**
A：不是。每个 skill 独立可用。调度器只是建议最佳顺序，你可以跳过任何环节。

**Q：没有 PRD 能直接用 do-eng 吗？**
A：可以。do-eng 会尽量从代码推断上下文，但有 PRD 效果更好。

**Q：do-dev 会自动执行所有子技能吗？**
A：不会。do-dev 是半自动的，列出建议后等你确认才执行。

**Q：CHANGELOG 文件名必须是英文吗？**
A：不必须。所有 skill 通过模式匹配查找，支持 `变更日志.md`、`CHANGELOG.md` 等各种命名。

**Q：什么是 context rot？为什么需要上下文工程？**
A：随着 AI 的上下文窗口被填满，输出质量会逐步劣化。do-dev 通过让子 skill 在独立的全新上下文中执行来解决这个问题，主调度器保持精简（~30-40% 利用率）。

**Q：原子 Git 提交有什么好处？**
A：每个任务独立提交，可以用 `git bisect` 精确定位引入 bug 的任务，可以单独 revert 某个任务而不影响其他任务，git 历史更清晰可读。

**Q：Discussion 和 Research 阶段可以跳过吗？**
A：可以。用户说"用默认"跳过 Discussion，说"跳过调研"跳过 Research。但建议在非配置类变更时至少走 Research。

**Q：do-dev 的恢复模式怎么用？**
A：do-dev 在 `.do-dev/state.json` 中持久化进度。下次启动时加 `--resume`，会从中断的阶段继续。自动模式下每个阶段还会保存 patch 检查点，可手动回退。
