# 多会话并行开发规范

**版本:** v1.0 | **日期:** 2026-03-21

---

## 一、问题与方案

### 1.1 问题

用户的工作模式是同时开多个 Claude Code 会话，每个会话处理不同功能方向：

```
会话 A: info-feed 去重优化    → up-prd → do-design → do-eng
会话 B: info-feed 简报改版    → up-prd → do-eng → do-qa
会话 C: info-feed RSS 扩展    → do-eng（直接实现）
```

在单一 git 分支上同时操作会导致：
- **文件写冲突**：后写覆盖先写
- **状态混乱**：无法区分哪个功能正在由谁处理
- **版本号竞争**：多个会话同时 bump 版本
- **依赖误判**：读到其他功能的完成状态，误以为自己的前序已完成

### 1.2 解决方案：双层隔离

```
┌─────────────────────────────────────────────────┐
│  代码隔离层：Git Worktree（每个 feature 独立分支）  │
│  ┌─────────────────────────────────────────────┐ │
│  │ 状态隔离层：.features/（按 feature 独立状态） │ │
│  └─────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
```

- **Git Worktree**：解决代码冲突 — 每个 feature 在独立分支+独立目录中开发
- **`.features/`**：解决状态可见性 — 所有会话通过注册表互相感知

## 二、Git Worktree 规范

### 2.1 什么是 Worktree

Git worktree 允许在同一个仓库中同时检出多个分支到不同目录。它们共享同一个 `.git` 数据库（磁盘开销极小），但拥有独立的工作目录。

```
claudecode_workspace/                    ← main 分支（主工作区）
├── .claude/worktrees/
│   ├── feat-info-feed-dedup/            ← 分支 feat/info-feed-dedup
│   └── feat-info-feed-briefing/         ← 分支 feat/info-feed-briefing
```

### 2.2 三种使用方式

**方式一：`claude -w` 启动会话（推荐）**

```bash
# 启动一个隔离会话，自动创建 worktree + 分支
claude -w feat-info-feed-dedup

# 搭配 tmux（自动创建 tmux session）
claude -w feat-info-feed-dedup --tmux
```

适用场景：用户手动开多个终端，每个终端处理一个功能。

**方式二：Agent isolation: worktree（do-dev 内部使用）**

```
# do-dev 调度子 skill 时使用
Agent(
  prompt="执行 do-eng...",
  isolation: "worktree"
)
```

适用场景：do-dev 在同一会话中调度子 skill，需要代码隔离。

**方式三：手动管理**

```bash
# 创建
git worktree add .claude/worktrees/feat-info-feed-dedup -b feat/info-feed-dedup

# 在 worktree 中工作
cd .claude/worktrees/feat-info-feed-dedup

# 完成后合并
git checkout main
git merge feat/info-feed-dedup

# 清理
git worktree remove .claude/worktrees/feat-info-feed-dedup
git branch -d feat/info-feed-dedup
```

适用场景：需要精细控制分支策略时。

### 2.3 分支命名规范

```
feat/<project>-<feature-name>
```

示例：
- `feat/info-feed-dedup` — info-feed 去重优化
- `feat/info-feed-briefing-ui` — info-feed 简报 UI 改版
- `feat/doc-driven-dev-worktree` — 文档驱动开发工具的 worktree 支持

### 2.4 Worktree 目录规范

统一存放在 `.claude/worktrees/` 下，已加入 `.gitignore`。

```
.claude/worktrees/
├── feat-info-feed-dedup/          ← 目录名 = 分支名（/ 替换为 -）
└── feat-info-feed-briefing-ui/
```

### 2.5 合并策略

```
功能完成 → git checkout main → git merge feat/xxx → 解决冲突 → 清理 worktree
```

原则：
- **先合并改动大的分支**，小的分支 rebase 更容易
- 合并前运行测试/验证
- 合并后立刻清理 worktree（`git worktree remove`）

### 2.6 注意事项

| 坑 | 解法 |
|----|------|
| 端口冲突 | 每个 worktree 的 dev server 用不同端口 |
| SQLite 文件锁 | 数据库文件不在 git 中管理；或每个 worktree 用独立 db 实例 |
| 依赖安装 | 首次进入 worktree 时运行 `pip install` / `npm install` |
| 自动提交冲突 | worktree 分支完成前，暂停 auto-sync（或只对 main 做 auto-sync） |

## 三、`.features/` 状态管理

### 3.1 目录结构

```
project/.features/
├── _registry.md              ← 全局注册表（所有 feature 一览）
├── info-feed-dedup/
│   ├── status.md             ← 该功能的全生命周期状态
│   └── state.json            ← do-dev 调度状态（检查点）
└── info-feed-briefing/
    ├── status.md
    └── state.json
```

### 3.2 `_registry.md` — 全局注册表

所有会话读写同一个注册表文件。格式：

```markdown
# Feature Registry

| feature-id | version | status | active-skill | heartbeat | branch | 描述 |
|------------|---------|--------|-------------|-----------|--------|------|
| info-feed-dedup | v3.2 | active | do-eng | 2026-03-21T15:12 | feat/info-feed-dedup | 信息流去重优化 |
| info-feed-briefing | v3.3 | active | do-design | 2026-03-21T15:10 | feat/info-feed-briefing | 简报 UI 改版 |
| info-feed-rss | v3.4 | completed | — | 2026-03-21T14:00 | — | RSS 源扩展（已合并） |
```

字段说明：
- **feature-id**：功能标识（kebab-case）
- **version**：预留的 PRD 版本号
- **status**：`active` / `completed` / `abandoned`
- **active-skill**：当前正在执行的 skill（up-prd / do-design / do-eng / do-qa / —）
- **heartbeat**：最后一次状态更新的时间戳
- **branch**：Git 分支名（completed 后清空为 —）

### 3.3 `status.md` — Feature 流水线状态

每个 feature 独立维护一份完整的流水线状态：

```markdown
# Feature: info-feed-dedup

## Pipeline

| phase | status | skill | started | completed | note |
|-------|--------|-------|---------|-----------|------|
| prd | ✅ 已完成 | up-prd | 14:30 | 14:50 | PRD v3.2 已更新 |
| design | 🔄 进行中 | do-design | 14:55 | — | 设计审计阶段 |
| eng | ⏳ 待处理 | — | — | — | 等待 design |
| qa | ⏳ 待处理 | — | — | — | 等待 eng |

## Engineering Tasks（do-eng 阶段展开）

| task-id | name | status | wave | commit | note |
|---------|------|--------|------|--------|------|
| T-01 | 添加去重索引 | ✅ 已完成 | Wave 1 | abc1234 | — |
| T-02 | 去重检测逻辑 | 🔄 进行中 | Wave 2 | — | — |
| T-03 | 去重 API 端点 | ⏳ 待处理 | Wave 2 | — | 依赖 T-02 |

## QA Items（do-qa 阶段展开）

| item-id | name | status | severity | commit | note |
|---------|------|--------|----------|--------|------|
| BUG-01 | 去重误判 | ✅ 已修复 | critical | def5678 | — |
```

### 3.4 状态标记定义

| 标记 | 含义 | 写入时机 |
|------|------|---------|
| ⏳ 待处理 | 已规划未开始 | feature 创建时 |
| 🔄 进行中 | 正在执行 | skill 开始执行时 |
| ✅ 已完成 | 执行完成 | skill 完成时 |
| ❌ 失败 | 执行失败 | 出错时 |
| ⏸️ 暂停 | 等待用户/阻塞 | 需要人工介入时 |

### 3.5 Heartbeat 与孤儿检测

**不需要后台守护进程**。每个 skill 启动时执行惰性检测：

```
读取 _registry.md
对于每个 status == active 的 feature：
  如果 heartbeat > 30 分钟前：
    → 警告用户："Feature X 已 30+ 分钟无心跳，上次活跃 skill: Y"
    → 选项：
      A) 认领继续（更新 heartbeat）
      B) 标记为 abandoned
      C) 忽略
```

Heartbeat 更新时机：
- skill 开始执行时
- 每完成一个子步骤时（如 do-eng 每完成一个 Task）
- skill 完成时（同时更新 status）

### 3.6 版本号预留机制

up-prd 创建 feature 时：

```
1. 读取 _registry.md 中所有 version 列
2. 找最高版本号（包括 abandoned 的）
3. 预留下一个版本号 → 写入注册表
4. 该版本号不回收（即使 feature 被 abandoned）
```

### 3.7 依赖检查改造

**核心原则：每个 skill 只检查自己 feature 的 status.md**

```
do-eng 启动时：
  ✅ 正确：读 .features/info-feed-dedup/status.md → Pipeline.design == ✅
  ❌ 错误：读 DESIGN.md 的运行状态 → 那可能是另一个 feature 的状态
```

## 四、完整工作流示例

### 4.1 单功能完整流程

```bash
# 1. 启动隔离会话
claude -w feat-info-feed-dedup

# 2. 会话内执行
> /up-prd 信息流去重准确率太低，37% 被误分类
  → 创建 .features/info-feed-dedup/
  → 注册到 _registry.md
  → 更新 PRD.md

> /do-dev
  → Discussion → Research
  → 调度 do-design（Agent, isolation: worktree）
  → 调度 do-eng（Agent, isolation: worktree）
  → 调度 do-qa（Agent, isolation: worktree）
  → 更新 status.md 各阶段状态

# 3. 完成后合并
> 合并到 main，清理 worktree
  → _registry.md 标记为 completed
```

### 4.2 多功能并行

```
终端 1                          终端 2                          终端 3
claude -w feat-dedup            claude -w feat-briefing         claude -w feat-rss
│                               │                               │
├ /up-prd 去重优化              ├ /up-prd 简报改版              ├ /do-eng RSS 扩展
├ /do-dev                       ├ /do-dev                       ├ 直接写代码
├   → do-design                 ├   → do-eng                    ├ 原子 commit
├   → do-eng                    ├   → do-qa                     ├ 合并到 main
├   → do-qa                     ├ 合并到 main                   └ ✅ done
├ 合并到 main                   └ ✅ done
└ ✅ done
```

每个终端在独立 worktree 中工作，互不干扰。通过 `.features/_registry.md` 互相感知状态。

## 六、.gitignore 配置

需要在项目根目录 `.gitignore` 中添加：

```
# === Worktree 工作目录（每个 feature 的临时代码副本）===
.claude/worktrees/

# === Feature 状态文件（可选：如果不想提交到远程）===
# .features/
```

建议 `.features/` 不加入 gitignore — 状态文件应该提交到 git，这样多台设备同步时也能看到 feature 状态。
