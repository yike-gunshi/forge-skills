"---\nname: forge-ship\ndescription: Use when a reviewed feature or fix branch in a worktree needs to be shipped through PR into the remote base branch and, when requested, synchronized back to the local base-branch worktree.\nallowed-tools:\n  - Bash\n  - Read\n  - Write\n  - Edit\n  - Grep\n  - Glob\n  - AskUserQuestion\n---\n\n> **文档落地路径**：遵循 forge-doc-policy 规范。完整白名单 + frontmatter schema 见\n> `~/claudecode_workspace/工具/forge-cookbook/skills/forge-doc-policy/doc-paths.md`。\n\n# /forge-ship：发布上线\n\n## 核心原则\n\n`ship` 不是“创建 PR 就结束”。它有两个模式：\n\n- **PR only**：提交、推送、创建或更新 PR，然后停止。\n- **Full ship**：提交、推送、创建或更新 PR，检查 PR 状态，合并 PR 到远程基础分支，再把本地基础分支 worktree 同步到 `origin/<基础分支>`。\n\n当用户明确说“ship / 发布 / 合并 / 通过 PR / 更新远程和本地 main”时，默认使用 **Full ship**。当用户只说“创建 PR / 提交 PR / 先发 PR”时，使用 **PR only**。\n\n永远区分这些状态：代码已提交、分支已推送、PR 已创建、PR 已合并、远程 main 已更新、本地 main 已同步、ECS 已部署。除非真的执行部署并验证运行态，否则不要说线上已更新。\n\n---\n\n## 前置脚本\n\n每次先运行：\n\n```bash\n_BRANCH=$(git branch --show-current 2>/dev/null || echo \"unknown\")\necho \"当前分支: $_BRANCH\"\ngit status --short\ngit remote -v\n```\n\n如果当前分支就是基础分支（通常是 `main`），不要直接 ship。先询问用户是要：\n- A) 新建功能分支再 ship（推荐）\n- B) 只同步本地 main\n- C) 停止\n\n---\n\n## AskUserQuestion 格式规范\n\n每次提问结构：\n1. **重新聚焦**：当前项目、分支、当前步骤\n2. **通俗解释**：说清楚这个选择会改变什么\n3. **给出建议**：推荐选项 + 一句话原因\n4. **列出选项**：`A) B) C)`\n\n---\n\n## 第0步：确定 Ship 模式和基础分支\n\n先确定模式：\n\n```bash\n# 根据用户措辞判断；不确定时询问\n# PR only: 只创建/更新 PR\n# Full ship: PR 合并后同步本地基础分支\n```\n\n确定基础分支：\n\n```bash\n# 1. 如果已有 PR，读取 PR base\ngh pr view --json baseRefName -q .baseRefName 2>/dev/null\n\n# 2. 没有 PR，则读取仓库默认分支\ngh repo view --json defaultBranchRef -q .defaultBranchRef.name 2>/dev/null\n\n# 3. 都失败则回退到 main\n```\n\n记录：\n- 当前分支\n- 基础分支\n- ship 模式\n- 当前 HEAD\n- `origin/<基础分支>` HEAD\n\n---\n\n## 第1步：安全检查\n\n检查当前 worktree：\n\n```bash\ngit status --short --branch\ngit fetch origin <基础分支> --quiet\ngit diff origin/<基础分支> --stat\ngit log origin/<基础分支>..HEAD --oneline\n```\n\n如果有未提交更改，询问：\n- A) 将本次相关更改加入暂存区并提交（推荐）\n- B) 交互式选择文件\n- C) 停下来，让我处理\n\n不要自动加入无关文件。特别注意构建产物，如 `frontend-react/tsconfig.app.tsbuildinfo`。\n\n如果是 Full ship，提前定位本地基础分支 worktree，但此时不要修改它：\n\n```bash\n_BASE=<基础分支>\n_MAIN_CWD=$(git worktree list --porcelain | awk -v branch=\"refs/heads/${_BASE}\" '\n  BEGIN { RS=\"\"; FS=\"\\n\" }\n  $0 ~ \"branch \" branch {\n    for (i=1; i<=NF; i++) {\n      if ($i ~ /^worktree /) {\n        sub(/^worktree /, \"\", $i)\n        print $i\n      }\n    }\n  }\n')\necho \"$_MAIN_CWD\"\n```\n\n如果找不到本地基础分支 worktree，记录“未找到”，不要自动创建，除非用户明确要求。\n\n---\n\n## 第2步：同步基础分支到当前功能分支\n\n```bash\ngit fetch origin <基础分支> --quiet\ngit merge origin/<基础分支> --no-edit\n```\n\n如果项目约定使用 rebase，需先确认，不要擅自 rebase 已共享分支。\n\n如果有冲突：\n1. 列出冲突文件。\n2. 解释冲突发生在当前功能分支合并基础分支阶段。\n3. 帮助解决冲突。\n4. 解决后重新运行关键测试。\n5. 冲突未解决前停止 ship。\n\n---\n\n## 第3步：运行项目验证\n\n不要使用 `npm test || yarn test || ... || echo \"未找到测试命令\"` 这种链式命令，它会吞掉真实失败。\n\n先识别项目可用命令：\n\n```bash\ntest -f package.json && cat package.json\ntest -f frontend-react/package.json && cat frontend-react/package.json\ntest -f pyproject.toml && sed -n '1,160p' pyproject.toml\ntest -f pytest.ini && cat pytest.ini\n```\n\n优先运行与本次改动相关、项目历史已验证的命令。对于 info2action 常见组合：\n\n```bash\nINFO2ACTION_CLUSTER_BACKEND=sqlite uv run pytest tests/test_summary_writer.py -q\ncd frontend-react && npx vitest run <相关测试文件>\ncd frontend-react && npm run lint\ncd frontend-react && npm run build\npython3 -m py_compile <相关 Python 文件>\ngit diff --check\n```\n\n如果测试失败：\n1. 分析是否与本次改动相关。\n2. 相关则修复后重跑。\n3. 不相关则列出证据并询问是否继续。\n4. 未经确认不要跳过失败测试。\n\n如果没有可用测试，询问：\n- A) 为本次改动补基础测试（推荐）\n- B) 只做静态检查和人工验证\n- C) 停止\n\n---\n\n## 第4步：发布前 diff 审查\n\n```bash\ngit diff origin/<基础分支> --stat\ngit diff origin/<基础分支>\n```\n\n检查：\n- 是否有调试代码：`console.log`、`debugger`、临时 `print`\n- 是否有密钥、token、私有配置\n- 是否有 `TODO: 上线前删除`\n- 是否误提交构建产物、缓存、tsbuildinfo\n- 是否包含无关文件\n- 是否修改了数据库、prompt、skill、部署脚本等高风险区域\n\n发现问题先修复或询问，不要带病 push。\n\n---\n\n## 第5步：更新 CHANGELOG\n\n如果仓库存在 `CHANGELOG.md`，且本次变更值得未来追溯，则更新。\n\n```bash\ngit log origin/<基础分支>..HEAD --oneline --no-merges\n```\n\n如果只是纯机械变更，或用户明确不需要文档，允许跳过，但发布总结要写明“CHANGELOG 未更新”。\n\n如果存在 `VERSION` 文件，询问是否升级版本：\n- A) patch\n- B) minor\n- C) major\n- D) 不升级\n\n---\n\n## 第6步：最终提交\n\n暂存本次相关文件：\n\n```bash\ngit add <本次相关文件>\ngit status --short\n```\n\n确认无关文件未暂存后提交：\n\n```bash\ngit commit -m \"<type>: <summary>\"\n```\n\n如果已经有提交且只有 CHANGELOG/VERSION 变更，可单独提交：\n\n```bash\ngit add CHANGELOG.md VERSION\ngit commit -m \"chore: update changelog\"\n```\n\n提交后记录 commit SHA。\n\n---\n\n## 第7步：推送当前分支\n\n```bash\ngit push -u origin <当前分支>\n```\n\n如果推送被拒绝，不要直接 `git pull --rebase`。先查看远端差异：\n\n```bash\ngit fetch origin <当前分支> --quiet\ngit log --left-right --graph --oneline HEAD...origin/<当前分支>\n```\n\n询问：\n- A) rebase 到远端分支后再推送（推荐，若只是远端新增提交）\n- B) merge 远端分支后再推送\n- C) 停止，让我处理\n\n不要 force push，除非用户明确要求且确认影响范围。\n\n---\n\n## 第8步：创建或更新 PR\n\n先检查是否已有 PR：\n\n```bash\ngh pr view --json number,url,state,baseRefName,headRefName 2>/dev/null\n```\n\n如果没有 PR，创建：\n\n```bash\ngh pr create \\\n  --title \"<PR标题>\" \\\n  --body \"<PR描述>\" \\\n  --base <基础分支> \\\n  --head <当前分支>\n```\n\nPR 描述必须包含：\n- 变更摘要\n- 测试情况\n- 注意事项\n- 是否需要部署\n- 是否有 migration / 数据修正 / prompt 变更\n\nPR 创建后记录 PR URL。\n\n如果模式是 **PR only**，到这里停止，并输出当前状态。\n\n---\n\n## 第9步：PR 合并前检查\n\nFull ship 必须检查 PR 是否可以合并：\n\n```bash\ngh pr view <PR编号> --json number,url,state,isDraft,mergeable,mergeStateStatus,statusCheckRollup,reviewDecision,baseRefName,headRefName\n```\n\n规则：\n- `state` 必须是 `OPEN`。\n- `isDraft` 必须是 `false`。\n- `mergeable=CONFLICTING` 时停止。\n- `mergeable=UNKNOWN` 时等待片刻并重查；仍 UNKNOWN 则说明 GitHub 尚未计算完成，不要强行合并。\n- `statusCheckRollup` 为空时，只能说“无远程 checks”，不能说“CI 已通过”。\n- 有 failed check 时停止并列出失败项。\n- 有 pending check 时优先等待：\n\n```bash\ngh pr checks <PR编号> --watch\n```\n\n如果 `reviewDecision` 表示需要 review，按仓库规则停止或询问用户，不绕过保护规则。\n\n---\n\n## 第10步：合并 PR 以更新远程基础分支\n\n确认 PR 可合并后，根据仓库约定选择 merge 方式。\n\n默认推荐 merge commit，因为它保留分支提交历史，并且本地基础分支若曾快进到功能分支，也更容易继续快进到远程 merge commit：\n\n```bash\ngh pr merge <PR编号> --merge --delete-branch=false\n```\n\n如果仓库约定 squash 或 rebase，先确认：\n\n```bash\ngh pr merge <PR编号> --squash --delete-branch=false\ngh pr merge <PR编号> --rebase --delete-branch=false\n```\n\n合并失败时：\n1. 输出失败原因。\n2. 不绕过 review、CI、权限或保护规则。\n3. 不 force push。\n4. 停在 PR 合并阶段。\n\n合并后：\n\n```bash\ngit fetch origin <基础分支> --quiet\ngit rev-parse origin/<基础分支>\n```\n\n记录远程基础分支新 SHA。\n\n---\n\n## 第11步：同步本地基础分支 worktree\n\n只从 `origin/<基础分支>` 同步本地基础分支。不要用功能分支直接覆盖本地 main。\n\n如果第1步没找到本地基础分支 worktree，重新查找：\n\n```bash\n_BASE=<基础分支>\n_MAIN_CWD=$(git worktree list --porcelain | awk -v branch=\"refs/heads/${_BASE}\" '\n  BEGIN { RS=\"\"; FS=\"\\n\" }\n  $0 ~ \"branch \" branch {\n    for (i=1; i<=NF; i++) {\n      if ($i ~ /^worktree /) {\n        sub(/^worktree /, \"\", $i)\n        print $i\n      }\n    }\n  }\n')\n```\n\n如果仍找不到：\n- 报告“未找到本地 `<基础分支>` worktree”。\n- 不自动创建。\n\n如果找到，先检查：\n\n```bash\ngit -C \"$_MAIN_CWD\" fetch origin \"$_BASE\" --quiet\ngit -C \"$_MAIN_CWD\" status --short --branch\ngit -C \"$_MAIN_CWD\" rev-list --left-right --count \"origin/${_BASE}...HEAD\"\n```\n\n处理规则：\n- 有未提交变更：列出文件，询问，不自动覆盖。\n- 本地基础分支有未推送提交或与远程分叉：停止并解释，不自动 reset。\n- 干净且可快进：执行\n\n```bash\ngit -C \"$_MAIN_CWD\" merge --ff-only \"origin/${_BASE}\"\n```\n\n如果用户选择 stash：\n- 必须使用带名字的 stash。\n- 如有 untracked 文件，说明是否包含 `-u`。\n- stash 后再同步。\n- 同步完成后不要自动 pop，除非用户明确要求。\n\n禁止：\n- `git reset --hard`\n- `git checkout --`\n- 静默覆盖用户未提交文件\n\n同步后验证：\n\n```bash\ngit -C \"$_MAIN_CWD\" rev-parse HEAD\ngit rev-parse \"origin/${_BASE}\"\n```\n\n两者一致，才能说“本地基础分支已同步到远程基础分支”。\n\n---\n\n## 第12步：发布完成总结\n\n总结必须拆状态：\n\n```text\n+====================================================+\n|                  Ship Summary                      |\n+====================================================+\n| 模式：PR only / Full ship                           |\n| 功能分支：<branch>                                  |\n| 基础分支：<base>                                    |\n| 提交：<commit sha>                                  |\n| PR：<url>                                           |\n| PR 状态：已创建 / 已合并 / 未合并                    |\n| 远程基础分支：<origin/base sha>                      |\n| 本地基础分支 worktree：<路径 / 未找到>               |\n| 本地基础分支状态：已同步 / 未同步 / 因脏工作区暂停    |\n| 测试：通过 / 跳过 / 失败                             |\n| CHANGELOG：已更新 / 未更新                           |\n| 部署：未执行 / 已执行并验证                          |\n+====================================================+\n```\n\n如果没有执行 ECS 部署，必须写：\n\n```text\nECS/线上部署未执行。本次 ship 只表示代码进入远程基础分支，并同步了本地基础分支。\n```\n\n---\n\n## 重要规则\n\n- **先 review，后 ship**：`forge-ship` 默认用于 review 后的发布，不替代完整 code review。\n- **先同步基础分支，后发布**：功能分支必须先吸收最新 `origin/<基础分支>`。\n- **测试失败不发布**：除非用户明确确认失败与本次改动无关。\n- **不要假绿**：禁止用会吞失败的链式测试命令。\n- **PR 是远程 main 的入口**：需要进入远程 main 的代码必须通过 PR merge。\n- **本地 main 以 origin/main 为权威**：PR 合并后再同步本地 main。\n- **保护用户未提交变更**：本地 main 有脏文件时，不自动 reset、checkout 或覆盖。\n- **发布不等于部署**：ship 不代表 ECS 已更新；部署要单独执行并验证。\n- **不要 force push**：除非用户明确要求，并且当前在功能分支上。\n"
