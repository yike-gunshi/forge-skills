# 复盘 — 协作知识沉淀与双向诊断

## 输出位置

统一保存到 `~/claudecode_workspace/复盘/{工作方向}/` 目录：
- **工作方向**由 AI 根据本次工作内容自动判断（如 `info-feed`、`doc-driven-dev`、`design`），用户可覆盖
- 文件名格式：`{YYYY-MM-DD}-{HHMM}-[任务标签]-{主题关键词}.md`
  - 日期在最前，后跟 24 小时制时分（取复盘文档创建时的当前时间），便于按时间排序
  - 任务标签应与具体功能点/任务挂钩，示例：`[调整前端布局]`、`[增加数据来源]`、`[多数据源聚合]`、`[单次提交抓取]`、`[手动提交URL]`、`[服务端搜索升级]`
- 示例路径：`~/claudecode_workspace/复盘/info-feed/2026-03-22-1430-[多数据源聚合]-RSS-HN-Reddit-GitHub信息源扩展.md`

## 前置脚本（每次先运行）

```bash
_ROOT=$(git rev-parse --show-toplevel 2>/dev/null || pwd)
_BRANCH=$(git branch --show-current 2>/dev/null || echo "unknown")
echo "当前分支: $_BRANCH"
echo "项目根目录: $_ROOT"

# === Git 卫生检查 ===
git fetch --quiet 2>/dev/null
_AHEAD=$(git rev-list @{u}..HEAD --count 2>/dev/null || echo "?")
_BEHIND=$(git rev-list HEAD..@{u} --count 2>/dev/null || echo "?")
echo "远程同步: 领先 $_AHEAD / 落后 $_BEHIND"
if [ "$_BEHIND" != "?" ] && [ "$_BEHIND" -gt 0 ]; then
  echo "⚠️ 落后远程 $_BEHIND 个提交，自动 rebase..."
  _HAS_CHANGES=$(git status --porcelain 2>/dev/null | head -1)
  [ -n "$_HAS_CHANGES" ] && git stash --quiet 2>/dev/null && _STASHED=1 || _STASHED=0
  git pull --rebase --quiet 2>/dev/null && echo "✅ 已同步远程" || echo "❌ rebase 失败，请手动处理"
  [ "$_STASHED" = "1" ] && git stash pop --quiet 2>/dev/null
fi
_DIRTY=$(git status --porcelain 2>/dev/null | head -5)
[ -n "$_DIRTY" ] && echo "⚠️ 工作区有未提交改动"
```

## 文档结构

### 一、项目概览（200字以内）

分三点，每点一句话：
- **我的需求**：当时想解决什么问题
- **AI 做了什么**：交付了哪些模块/功能
- **最终交付**：实际跑起来的是什么

---

### 二、知识地图

按知识领域分块（如"前端布局"、"数据库设计"、"交互模式"）。每个领域包含以下 **5 个模块**：

---

#### 模块 1：你遇到了什么问题

用用户自己的语言还原当时的困惑，不用术语。一段话即可。目的是下次读时能快速唤起记忆。

---

#### 模块 2：关键概念速查表

| 概念 | 一句话解释 | 什么时候用 | 下次告诉 AI 的关键词 |
|------|-----------|-----------|-------------------|

---

#### 模块 3：实践归因

记录"谁做了什么决定、为什么"，形成决策审计轨迹。

##### 我的决策

- [决策内容]
  - 背后的诉求：一句话还原真实动机

##### AI 的决策

- [决策内容]
  - 替代方案：方案A / 方案B
  - 选择理由：一句话
  - 潜在风险：什么情况下可能是错的

##### 最终定稿

- 最终方案：一句话
- 与初始方案的差异：经历了哪些调整
- 当前稳定性：可靠 / 仍有隐患

---

#### 模块 4：最佳实践

每条实践用以下结构：

```
▍ [实践名称]
- 做法：一句话说清楚怎么做
- 为什么：一句话说清楚收益
- 反模式：什么情况下不该用，或用错了会怎样
```

附 2-3 条最有价值的参考资源（不强制分类，禁止裸链接）：
- [资源名] — 解决了什么问题，看哪部分

---

#### 模块 5：避坑指南

| 坑 | 触发条件 | 解决方案 |
|----|---------|---------|

---

### 三、效率词典

| 中文 | 英文 | 一句话解释 | 场景 | 下次怎么说 |
|------|------|-----------|------|-----------|

用户下次遇到类似需求时，直接查这张表找到关键词。

---

### 四、下次协作指南

不给通用模板，只给**基于本次问题的具体改进**：

- **关键决策点**：哪些地方必须自己决定，不能全交给 AI（分点列出）
- **验收标准前置**：基于本次返工经验，下次开始前应先对齐哪几个验收点
- **沟通策略**：基于第六章诊断结果，给出 2-3 条具体的表达改进建议

---

### 五、AI 表现复盘

#### 评分总览

| 维度 | 评级 | 说明 |
|------|------|------|
| 需求理解准确度 | 🟢/🟡/🔴 | 一句话 |
| 自主决策质量 | 🟢/🟡/🔴 | 一句话 |
| 迭代效率 | 🟢/🟡/🔴 | 一句话 |
| 输出完整度 | 🟢/🟡/🔴 | 一句话 |

#### 具体问题

每个问题用以下结构：

- **问题标题**
  - 表现：发生了什么
  - 根因：AI 侧的具体失误（误解/遗漏/过度/低效）
  - 浪费轮次：N 轮
  - 应有做法：AI 当时应该怎么做

---

### 六、用户表达复盘

#### 评分总览

| 维度 | 评级 | 说明 |
|------|------|------|
| 需求精确度 | 🟢/🟡/🔴 | 一句话 |
| 上下文完整度 | 🟢/🟡/🔴 | 一句话 |
| 拆分粒度 | 🟢/🟡/🔴 | 一句话 |

#### 模糊描述诊断

| 你说的 | AI 理解成了 | 精确说法 | 关键词 |
|--------|------------|---------|--------|

#### 遗漏的关键上下文

| 你以为 AI 知道的 | 实际情况 | 影响 | 下次应该说 |
|-----------------|---------|------|-----------|

#### 可以更快达到目的的做法

限 3-5 条，必须是本次实际可用的建议，分点列出。

---

## 评级标准

全文档统一：
- 🔴 = 同一问题返工 ≥3 轮，或方向性错误
- 🟡 = 需要 1-2 轮修正，或有遗漏但不影响交付
- 🟢 = 一次到位或决策合理

## 执行流程

1. **收集上下文**：读取项目文件结构、git log、docs/ 目录下已有文档
2. **回溯对话**：找出用户表现出不确定、反复修改、或 AI 理解偏差的节点
3. **识别决策点**：区分用户明确指令 vs AI 自主决策
4. **调研补充（Research）**：
   - 对每个知识领域做 web search，查找行业最佳实践
   - 对用户描述模糊的点，调研该领域的标准术语和常见表达方式
   - 对用户可能不知道的领域，补充"这个领域还有哪些值得了解的实践"
   - 调研结果直接融入各模块（概念速查表、最佳实践、效率词典），不单独成章
5. **组织输出**：按上述框架生成文档
6. **保存**：写入 `~/claudecode_workspace/复盘/{工作方向}/` 目录
7. **生成内容图表并截图保存**（关键步骤）：
   - **判断图表需求**：根据本次工作内容，决定需要哪些图表来结构化展示复盘。常见图表类型：
     - 架构图 / 流程图（适用于系统设计、模块交互）
     - 前后对比图（适用于重构、UI 改造）
     - 时间线图（适用于多阶段任务、问题排查链路）
     - 数据流图（适用于数据管线、多源聚合）
     - 评分雷达图（固定：AI/用户双方评分可视化）
   - **生成流程**（每张图表重复）：
     1. 用 `show-widget` 生成 SVG/HTML 图表（此时图表会在对话中渲染显示）
     2. 调用 `codepilot_load_widget_guidelines` 获取设计规范（首张图前调用一次即可）
     3. 将 widget HTML 写入临时文件 `/tmp/fupan_chart_N.html`
     4. 用 `render_widget.js` 截图保存为 PNG：
        ```bash
        NODE_PATH=$(npm root -g 2>/dev/null || echo "/opt/homebrew/lib/node_modules") \
        node ~/claudecode_workspace/.claude/skills/render_widget.js \
        /tmp/fupan_chart_N.html \
        ~/claudecode_workspace/复盘/{工作方向}/assets/{日期}-{主题}_{图表名}.png
        ```
     5. 在复盘文档中插入图片引用：`![图表名](assets/{日期}-{主题}_{图表名}.png)`
   - **注意**：assets 目录需要提前创建（`mkdir -p`）
8. **回流 Memory**（关键步骤）：
   - 读取当前 MEMORY.md（Claude Code 自动记忆目录下）
   - 从本次复盘中提取需要跨会话记住的内容，按以下规则分类：
     - **Feedback（行为指令）**：本次踩的坑 → "下次做X/不做Y"，一行描述 + 详细文件
     - **Project（项目状态）**：架构变更、活跃设计方向 → 更新已有条目或新建
     - **Reference（索引指向）**：在 MEMORY.md 的 Reference 区更新复盘文件计数和关键词
   - 写入规则：
     - 每条 Feedback 创建独立 `.md` 文件（frontmatter 含 name/description/type），在 MEMORY.md 加一行索引
     - 已有相同主题的 memory → **更新**而非新建，避免重复
     - 只提取"做X/不做Y"的行为指令，**不复制**复盘文件的完整内容
     - MEMORY.md 总行数控制在 40 行以内（超过 200 行会被截断，留足余量）
   - 输出：列出本次新增/更新的 memory 条目，让用户确认
9. **更新 WORKSPACE-PULSE**（关键步骤）：
   - 读取 `~/claudecode_workspace/WORKSPACE-PULSE.json`
   - 基于本次复盘内容，**增量更新**以下字段：
     - `active_work`：更新对应项目的 status/focus/context/blockers；新项目则追加；已完成的项目保留但 status 标记为"已完成"
     - `recent_learnings`：追加本次复盘中的新知识（topic + insight + source 指向复盘文件路径），不设上限，不删除历史条目
     - `pain_points`：新增本次发现的痛点（含 severity/attempted_solutions/still_unsolved）；已解决的痛点从列表中移除
     - `itch_list`：新增"想做但还没做"的想法；已启动的移到 active_work
     - `unsolved_questions`：新增悬而未决的设计/架构问题；已解决的移除
     - `content_pipeline`：如果涉及内容创作，更新 topic/stage/angle
     - `tools_evaluated`：如果评估了新工具，追加 tool/verdict/keep_watching
     - `investment_watch`：如果发现新的投资/趋势方向，追加
   - 更新 `updated_at` 为当前时间，`updated_by` 为 `fupan@{本次复盘主题}`
   - 写回文件
10. **生成复盘仪表盘并截图保存**（关键步骤）：
    - 用 `show-widget` 渲染一个**复盘仪表盘**（此时仪表盘会在对话中渲染显示）
    - 仪表盘包含以下区块（从上到下）：
      1. **标题栏**：复盘主题 + 日期 + 工作方向标签
      2. **项目概览**：需求 → 交付，三句话版本（直接取文档"一、项目概览"内容）
      3. **评分双栏**：左栏 AI 评分（四维度 + 颜色标记），右栏用户评分（三维度 + 颜色标记）
      4. **关键收获**：从知识地图中提取 top 3 最有价值的实践/避坑，一行一条
      5. **Memory 更新**：本次新增/更新的 memory 条目列表
      6. **底栏**：复盘文件完整路径 + git 推送状态
    - 仪表盘设计原则：
      - 信息密度高，一屏看完核心内容
      - 评分用 🟢🟡🔴 颜色编码，一眼看出好坏
    - **截图保存**（与 Step 7 相同流程）：
      1. 将仪表盘 HTML 写入 `/tmp/fupan_dashboard.html`
      2. 用 `render_widget.js` 截图保存为 PNG：
         ```bash
         NODE_PATH=$(npm root -g 2>/dev/null || echo "/opt/homebrew/lib/node_modules") \
         node ~/claudecode_workspace/.claude/skills/render_widget.js \
         /tmp/fupan_dashboard.html \
         ~/claudecode_workspace/复盘/{工作方向}/assets/{日期}-{主题}_dashboard.png
         ```
      3. 在复盘文档末尾插入"七、复盘仪表盘"章节，包含图片引用：
         ```markdown
         ### 七、复盘仪表盘
         ![复盘仪表盘](assets/{日期}-{主题}_dashboard.png)
         ```
11. **提交与推送**：
    ```bash
    # 提交复盘文件（含图表+仪表盘）、memory 更新和 PULSE
    git add ~/claudecode_workspace/复盘/ 2>/dev/null
    git add ~/.claude/projects/*/memory/ 2>/dev/null
    git add ~/claudecode_workspace/WORKSPACE-PULSE.json 2>/dev/null
    git commit -m "docs: 复盘 — [主题关键词]"
    git push origin "$_BRANCH" 2>/dev/null && echo "✅ 复盘已推送"
    ```
12. **最终输出**：
    - 在对话中重新用 `show-widget` 渲染仪表盘（确保对话末尾能看到完整概览）
    - 列出所有生成的图表文件路径
    - **注意**：此步骤的 widget 不需要再截图（Step 10 已保存），纯粹是为了对话末尾的可读性

## Memory 回流规范

### 目录
- Memory 文件位于 Claude Code 的 auto-memory 目录（系统 prompt 中会注入路径，通常为 `~/.claude/projects/{编码后的工作目录路径}/memory/`）
- MEMORY.md 是索引文件，每次对话自动加载；其他 `.md` 文件需要 AI 主动 Read

### 什么该写入 Memory
- ✅ 行为指令："做X/不做Y" + 原因（一句话）
- ✅ 项目活跃状态：当前架构、设计方向、待解决问题
- ✅ 外部资源指向：API 文档、监控地址、关键配置文件位置

### 什么不该写入 Memory
- ❌ 代码实现细节（看代码就行）
- ❌ 完整踩坑过程（已在复盘文件中）
- ❌ 一次性任务的临时状态
- ❌ 已过时的信息（如已完成的迁移）

### Memory 文件格式
```markdown
---
name: 名称
description: 一行描述（用于判断是否相关，要具体）
type: feedback | project | reference
---

行为指令或项目状态描述。

**Why:** 原因（通常是踩坑经历）
**How to apply:** 什么场景下适用
```

## 约束规范

### 图表双向同步铁律
- **所有 `show-widget` 必须在对话和复盘文档中同时可见，没有例外**
- 每个 show-widget 都必须走完整流程：对话渲染 → 截图 PNG → 插入文档
- 包括内容图表（Step 7）和仪表盘（Step 10），不允许"只在对话中展示"
- Step 12 的重渲染是唯一例外：它复用 Step 10 的截图，不需要再次保存

### 内容规范
- **不放代码片段**：用自然语言描述概念
- **结构化优先**：所有内容尽量分点论述，避免大段文字
- **表格从简**：表格只用于天然适合对比/查找的内容（速查表、诊断表），不用于表达因果关系或权衡
- **信息密度优先**：每个知识块以模块为单位，不强求固定行数

### AI 表现复盘规范
- 必须归因到 AI 侧的具体失误，不能写"双方沟通不够"这类模糊归因
- 每个问题必须写"应有做法"
- 如果 AI 表现全部合格，仍需列出"做得好但可以更好"的点

### 用户表达复盘规范
- 聚焦可改进的表达方式，不评判用户的技术能力
- "模糊描述诊断"必须给出精确替代说法和关键词
- "遗漏上下文"只记录确实影响了结果的遗漏
- "更快达到目的的做法"限 3-5 条，必须是本次实际可用的建议

### Research 规范
- 每个知识领域至少做一次 web search
- 用户描述模糊的地方，必须调研该领域的标准表达方式并告诉用户"下次可以这样说"
- 用户给的信息不够充分时，主动补充该领域的关键最佳实践
- 资源必须权威：官方文档、知名作者博客、高赞社区讨论，禁止低质量转载
- 调研结果融入各模块，不单独成章

### 最终定稿是锚点
- 经多轮迭代后初始方案和最终方案可能差很多，这个差异本身就是有价值的学习内容
