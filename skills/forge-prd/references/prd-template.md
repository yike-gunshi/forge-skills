# PRD 模板结构参考

## 标准 PRD 章节结构

一份完整的 PRD 应包含以下章节（按实际项目需要裁剪）：

```markdown
# 项目名称 — 产品需求文档 (PRD)

**版本:** vX.Y
**日期:** YYYY-MM-DD
**状态:** [需求整理 / 设计中 / 开发中 / 已上线]

---

## 迭代历史摘要

> 保留所有版本，最新版在最前面。每个版本列核心变更 + CHANGELOG 定位。

### vX.Y — YYYY-MM-DD
- [变更1简述] — 详见 §3.1.3、CHANGELOG vX.Y
- [变更2简述] — 详见 §4.1、CHANGELOG vX.Y
- **迭代交付说明**：
  - 前端：[涉及页面/组件、交互变化]
  - 后端：[涉及 API/数据模型、逻辑变化]
  - 设计：[配色/布局变化]（如适用）
  - 验收标准：[每个变更点的验收条件]
  - 受影响文件：[关键文件路径提示]
  - 关联变更：[改 A 需同步改 B]

### vX.Y-1 — YYYY-MM-DD
- [变更简述] — 详见 §X.X、CHANGELOG vX.Y-1

### v1.0 — YYYY-MM-DD
- 初始版本

---

## 一、产品定位
- 一句话描述产品是什么
- 核心价值：用户能从中获得什么

## 二、信息架构
- 数据层级关系
- 功能模块划分

## 三、核心功能详细设计
### 3.X 功能名称
#### 3.X.1 功能概述
#### 3.X.2 详细行为
#### 3.X.3 异常处理
- 内联标记规范：
  - **[需修复]** — bug 或行为错误
  - **[需改进]** — 体验可优化
  - **[需修改]** — 需求变更
  - **[vX.Y 新增]** — 本次迭代新增
  - **[vX.Y 修改]** — 本次迭代修改

## 四、数据模型
- 数据库表结构（SQL schema）
- API 端点列表

## 五、已知问题与改进计划
### P0 — 必须修复
### P1 — 体验优化
### P2 — 后续规划

## 六、技术约束
- 框架限制
- 第三方 API 约束
- 性能要求

## 七、交互流程图
- 关键用户操作的伪代码流程

## 八、配色方案（可选，前端项目需要）

## 九、前端设计系统（可选，前端项目需要）

## 十、技术栈

## 附录：修改清单总结
- 按优先级列出需执行的修改
```

## CHANGELOG 格式

CHANGELOG 文件名不固定（可能叫 PRD-CHANGELOG.md、CHANGELOG.md、变更日志.md 等），通过模式匹配定位。

```markdown
# PRD 变更日志

## [vX.Y] - YYYY-MM-DD

### 变更背景
[为什么要做这次变更？来自用户反馈、Bug 发现、还是新需求？]

### 用户原始需求
[用户的原话或需求描述]

### 设计方案
[采用的方案是什么？简要描述]

### 关键决策
| 议题 | 决定 | 原因 |
|------|------|------|
| [讨论点1] | [做/不做/推迟] | [为什么] |

### 影响范围
- 新增：[列出新增内容]
- 修改：[列出修改内容]
- 删除：[列出删除内容]

### PRD 变更清单
| 章节 | 变更类型 | 描述 |
|------|----------|------|
| 迭代摘要 | 新增条目 | vX.Y 摘要 |
| 三.X | 新增 | [新功能描述] |
| 五 | 更新 | [问题列表调整] |
```

## 新建 CHANGELOG 的流程

当项目没有 CHANGELOG 文件时，需要基于以下信息回溯生成：

1. **项目文档**：README、已有的版本说明等
2. **Git 历史**：`git log --oneline --since="项目创建日"` 识别关键里程碑
3. **PRD 版本信息**：如果 PRD 已有版本号，据此推断历史变更
4. **代码变更模式**：大的 commit 或 tag 通常对应功能迭代

回溯生成的历史记录标注 `[回溯记录]`，与后续实时记录区分。
