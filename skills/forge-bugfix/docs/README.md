# /do-bugfix 使用文档

## 定位

`/do-bugfix` 是 do-dev 生态中的 **Bug 调查与修复工具**，基于 gstack/investigate 的系统性调试方法论，中文化并适配了 do-dev 的状态管理体系。

### 与其他 skill 的关系

```
/up-prd（需求管理）
  → /do-dev（开发调度）
       ├── /do-design（设计）
       ├── /do-eng（工程实现）
       ├── /do-qa（QA 测试）        ← 发现 Bug
       └── /do-bugfix（Bug 修复）   ← 修复 Bug
```

| Skill | 职责 | 何时用 |
|-------|------|--------|
| `/do-qa` | 系统性测试 + 批量发现 Bug | 功能开发完成后，全面验证 |
| `/do-bugfix` | 单个 Bug 的深度调查 + 修复 | 用户报告具体问题时 |
| `/investigate` | 原版英文系统性调试 | 不需要 do-dev 生态集成时 |

**核心区别**：do-qa 是"主动出击找 Bug"，do-bugfix 是"被动接单修 Bug"。

---

## 使用场景

### 场景 1：用户直接报告 Bug

```
用户："点击提交按钮后页面白屏了"
→ /do-bugfix
```

### 场景 2：do-qa 发现了复杂 Bug

do-qa 的 WTF 启发式触发停止，或 Bug 根因复杂需要深度调查。

### 场景 3：线上 Bug 反馈

```
用户："线上有用户反馈搜索结果不准确"
→ /do-bugfix
```

### 场景 4：回归 Bug

```
用户："上次改完之后，之前好用的功能不行了"
→ /do-bugfix（会自动查 git log 找引入点）
```

### 场景 5：do-dev 调度

do-dev 可以在 QA 阶段后调用 do-bugfix 处理 do-qa 标记为 deferred 的 Bug。

## 执行流程详解

### 第0步：环境检测与问题理解

1. 检测可用的复现引擎（gstack/browse、Playwright、纯代码）
2. 检测项目框架和运行时
3. 检测本地运行的应用（扫描常用端口）
4. 收集用户的问题描述
5. 检查工作区状态（未提交变更 → 建议先 commit 或 stash）

### 第1步：根因调查

1. 收集症状：错误信息、堆栈跟踪、浏览器控制台错误、网络请求失败
2. 追踪代码：用 Grep 找引用，用 Read 理解逻辑，画数据流
3. 查最近变更：`git log` 检查受影响文件的历史
4. 复现 Bug：用浏览器或 API 调用确定性地触发 Bug

**产出**：`"根因假设：……"` — 具体的、可验证的断言

### 第1.5步：范围锁定

锁定编辑范围到受影响文件的最窄目录，防止越界修改。

### 第2步：模式分析

将 Bug 与 8 种已知模式对比：竞态、空值、状态污染、集成失败、配置漂移、缓存过期、前后端不一致、SQL 错误。检查历史修复记录，必要时搜索外部资源。

### 第3步：假设验证

在疑似根因处添加临时日志/断言，运行复现检查。**3 次失败自动停止**。

**红旗信号**：
- "先临时修一下" → 没有临时
- 还没追踪数据流就想修 → 在猜
- 每次修暴露新问题 → 错误层级

### 第4步：实现修复

1. 修根因，不修症状
2. 最小 diff + 原子提交
3. 写回归测试
4. 涉及 >5 个文件 → 必须用户确认

### 第5步：验证与报告

1. 全新复现确认已修复（截图对比）
2. 运行完整测试套件
3. 输出结构化调试报告
4. 更新 .features/ 状态、TODOS.md、QA.md

**报告格式**：
```
调试报告
════════════════════════════════════════
症状：     [用户看到的]
根因：     [实际问题]
修复：     [改了什么]
证据：     [测试输出]
回归测试： [测试文件位置]
状态：     完成 | 完成但有顾虑 | 阻塞
════════════════════════════════════════
```

## 三种复现引擎

| 引擎 | 适用场景 | 优势 |
|------|---------|------|
| **gstack/browse** | UI Bug、交互问题、样式问题 | 快速，带标注截图，命令式交互 |
| **Playwright** | 复杂 E2E 流程、API 拦截、前后端联动 | 可编程，网络请求拦截，多视口 |
| **纯代码** | 后端 Bug、数据问题、逻辑错误 | 无依赖，始终可用 |

引擎选择是自动的，优先级：gstack/browse > Playwright > 纯代码。

---

## 与 do-dev 生态的集成

- **.features/ 状态管理**：读取/写入 status.md 的 Bugfix Items 表
- **QA.md 联动**：修复 QA.md 中已知问题后更新状态
- **TODOS.md 联动**：标注已修复的 TODO，添加新发现的潜在问题

---

## 铁律

1. **不做根因分析就不写修复代码**
2. **3 次假设失败 → 自动停止**
3. **无法复现 → 不修**
4. **修复 >5 文件 → 必须确认**
5. **每个修复原子提交**
6. **修复后必须验证**
7. **可以随时升级（"我不确定"）**

## 来源

基于 gstack/investigate 的系统性调试方法论（Iron Law、Phase 1-5、3-strike rule、Pattern Analysis），中文化并新增：
- 三引擎复现架构（gstack/browse + Playwright + 纯代码）
- .features/ 状态管理集成
- QA.md / TODOS.md 联动
- do-dev 调度器协作
